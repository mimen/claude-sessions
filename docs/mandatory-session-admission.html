Managed session births

Artifact: /Users/mimen/Programming/Repos/claude-sessions/docs/mandatory-session-admission.html

Status: simplified Phase 1 is implemented, installed, and live-verified. CCS now records managed session births on the common launcher paths, blocks unmanaged nested persistent launches, and reports post-rollout integrity gaps without turning SessionStart into a brittle startup gate. The deferred strict admission controller remains out of scope.

Goal

Every persistent local Claude Code session is admitted or detected by CCS, receives creator and launch provenance with an explicit attestation source/strength, and declares whether it is independently user-managed or supporting work. Default views show user-managed work bodies and hide supporting sessions without deleting history.

Selected Phase 1 scope complete

Reframe CCS as a managed birth registry and integrity monitor, not a hostile-code admission controller. Make the correct path automatic for normal human, delegate, cmux, and scripted launches; infer native subagents using existing transcript evidence; retain unusual escapes as visible unclassified warnings. Harden only the bypasses that actually recur.
Keep

session_class, causal parent, creator_kind/ref, launch channel, familiar launcher shims, ccs delegate, default auxiliary hiding, and visible unclassified warnings.

Drop initially

Single-use claim tokens, fail-closed SessionStart, 1:N launch-event state machines, Unix-socket secret brokers, cross-hook native-Agent correlation, managed Agent SDK enforcement, cryptographic creator credentials, and exhaustive in-process succession.

Accept

Rare raw/SDK/in-process paths may appear as visible unclassified sessions and need manual repair. CCS protects against accidental and policy-compliant litter, not hostile code running as the same Unix user.

Phase 1 implementation contract

  1. Add typed birth metadata to the durable catalogue: creator_kind, creator_ref, launch_channel, and optional forked_from_session_id, alongside existing class and parent.
  2. Install a stable CCS Claude shim outside the updater-managed binary path. Inside cmux, accept its injected session ID/settings; outside cmux, generate the ID. Exec the dynamically resolved raw Claude binary by absolute path.
  3. Auto-register ordinary interactive new sessions as work_body + human. Resume/continue preserve existing birth metadata. Explicit persistent nested work must use ccs delegate or ccs session new. Keep the already-implemented --top-level/--child-of grammar in Phase 1; do not spend scope on renaming flags or compatibility aliases.
  4. Extend ccs delegate reservations with creator_kind=agent, creator ref, and launch channel while retaining existing parent/class/seat/route metadata and failure recording.
  5. Keep native Agent classification on existing sidechain/transcript-parent inference; do not add the four-hook admission choreography in Phase 1.
  6. Route CCS TUI new/fork and the known cmux, daemon, launchd, script, and eval callsites through managed launch commands where practical.
  7. Add a doctor/integrity report for post-cutoff unclassified sessions and preserve the existing visible warning behavior. Unknown edge paths remain repairable evidence, not startup failures.
  8. Keep SessionStart advisory/fail-open. Do not implement claim tokens, strict blocking, secret handoff, managed SDK enforcement, or process-succession machinery.
  9. Test launcher ordering inside/outside cmux, updater target changes, native/GPT provider environments, explicit session IDs, resume/continue, subcommands, no-persistence paths, delegate metadata, native-sidechain inference, TUI actions, and default visibility.

Long-term product semantics

The semantics below remain useful direction. Any strict enforcement mechanism that conflicts with the selected Phase 1 contract is deferred, not part of implementation authorization.

Managed coverage

Normal persistent launch paths register birth metadata through CCS.

Unintegrated raw, SDK, in-process, or future paths remain visible as unclassified warnings; Phase 1 does not brick Claude to enforce coverage.

Non-sessions

API calls and --no-session-persistence inference remain outside CCS admission.

They do not create local session history and therefore do not contribute to session litter.

Axes

Creator provenance and visibility/management class are separate.

work_body versus auxiliary answers management/default visibility; creator fields answer who caused the session.

Creator taxonomy

human | agent | automation.

Each non-human creator carries a stable reference: launching session for agents, job/daemon identity for automation.

Creator provenance

Store creator kind/ref and launch channel using honest best-effort evidence.

Phase 1 does not build cryptographic attestation or confidence tiers. Inferred/unknown origins stay distinguishable through channel metadata or a null creator.

Durable workers

Agent-created, user-managed workers are root work bodies.

They have no cost parent; creator provenance records the launching agent. Supporting runs remain auxiliary causal children.

Root authority

An explicit admission flag is sufficient for a non-human launcher to create a visible root.

No role-manifest allowlist or human approval token is required; correctness depends on mandatory intent declaration and policy/enforcement around the flag.

Automation auxiliaries

Parentless automation may create hidden auxiliary runs only with a mandatory paper trail.

The caller must provide a stable automation identity and a unique job/workflow run reference. parent_session_id remains reserved for session causality and cost rollup.

Admission grammar

Exactly one of three high-level admission forms is required.

--user-managed, --supporting-of <session|.>, or --automation-run <job-id> --run-ref <run-id>. Invalid combinations never reach reservation.

Native Agent tool

Preserve it using existing sidechain/transcript inference in Phase 1.

The documented multi-hook adapter remains available as future hardening, but current post-index parent/class inference already covers the visibility and cost-rollup need without brittle hook choreography.

Human convenience

Bare interactive human commands auto-admit as visible work bodies through CCS.

The familiar launcher names route through CCS. Print mode and non-human callers must use an explicit admission form.

Command surface

The familiar launcher names become CCS-aware executable shims.

claude, claude-native, and claude-gpt preserve current human UX through a dedicated PATH-precedent shim layer; updater-managed binaries and the existing shell function are migrated rather than overwritten in place.

Phase 1 enforcement

Automatic common-path registration plus visible integrity warnings.

The launcher shim and managed commands write metadata before launch. SessionStart remains advisory and fail-open; unknown sessions are reported rather than blocked.

Forks

CCS TUI/launcher forks prompt for intent, defaulting to visible independent work.

Fork lineage is stored separately when known. In-process branch/fork paths are best-effort or visibly unclassified in Phase 1.

Phase 1 lifecycle

Keep reservation/start/failure evidence without a full launch state machine.

Reuse the existing delegate-style metadata and never delete evidence or automatically retry work. Per-resume activation history is deferred.

Failure behavior

Phase 1 remains usable when CCS registration fails.

The session proceeds as visibly unclassified and doctor reports it. A dedicated break-glass subsystem is unnecessary until strict blocking exists.

Integrity warnings

Post-cutoff unmanaged sessions remain visible as unclassified warnings.

CCS retains them intact and does not guess parentage, silently hide them, archive them, or treat unknown provenance as managed.

Enforcement gaps

  • The global nested-launch guard is a Claude Code PreToolUse:Bash hook, not an operating-system admission boundary.
  • It blocks only simple direct print-mode commands. Wrappers, variables, compound shell, subprocesses, cmux, launchd, Agent tool, forks, and daemons bypass it.
  • The static vault scanner is manual and supports allow annotations; it is not runtime enforcement.
  • There is no trusted durable creator_kind, creator_ref, or launch_channel field today.

Verified Claude Code platform limits

  • SessionStart receives the real session ID after initialization. Exit code 2 and event-specific decision:block do not block it. Universal continue:false is documented across events, but its exact interactive/print/SDK termination behavior at SessionStart must be empirically verified before enforcement depends on it.
  • SessionStart also fires for resume, fork, clear, and compact. Fork can create a new identity inside an existing process; resume can switch identities; current docs do not explicitly settle whether clear changes the session ID. None can blindly replay the predecessor’s consumed launch token.
  • --bare skips hook auto-discovery; --safe-mode disables ordinary custom hooks while managed-policy hooks still apply. Both can remain persistent. Shims must reject unmanaged persistent use, and SDK/managed-policy integration must not rely solely on user/project settings.
  • Native Agent correlation is documented but split across events: Pre/PostToolUse include tool_use_id; Agent PostToolUse includes tool_response.agentId; SubagentStart/Stop include agent_id. SubagentStart remains context-only/non-blocking, so later events and integrity reconciliation must repair a failed birth write.
  • CCS can make accidental and policy-compliant bypasses fail reliably. It cannot be a hostile-code security boundary against arbitrary code running as the same Unix user unless the process is additionally sandboxed.

Deferred strict-mode decisions

These do not block the selected Phase 1 registry/integrity implementation.

  1. The exact typed schema and CLI flags for automation identity and unique job/workflow-run references.
  2. In-process succession semantics for /branch, /fork, /resume, conversation recovery, and potentially /clear, where no outer launcher can mint a fresh token or prompt for intent.
  3. Agent SDK enforcement for persistent sessions when embedding hosts disable filesystem settings with settingSources: [].
  4. CCS outage behavior: distinguish affirmative invalid admission from unavailable/corrupt claim infrastructure, and define a database-independent human break-glass path.
  5. Human adoption and audited repair semantics for auxiliaries, bypasses, and violations without rewriting immutable birth history.

Deferred strict-mode implementation direction

  1. Separate immutable session birth/origin from a 1:N stream of process launches and in-process activations. A resume or reactivation creates an admission/activation event without rewriting creator, class, parent, or lineage.
  2. Add typed session-origin fields: management class, creator kind/ref, creator-attestation source and strength, automation origin where applicable, launch channel, fork/clear lineage, and immutable creation timestamp.
  3. Add a dedicated admissions table with reservation ID, expected/new/existing session mode, token verifier or protected handoff reference, expiry, claim evidence, process/substrate identity, lifecycle state, and terminal reason.
  4. Require exactly one high-level intent: --user-managed, --supporting-of <session|.>, or parentless --automation-run <job-id> --run-ref <run-id>. Creator identity is derived independently; automation credentials require --run-ref as supplementary metadata on every persistent intent.
  5. Enforce parent authority: agent-created support may target only its current admitted session; automation may target only an active authorized session; an interactive human may explicitly override with an audited reason. Never accept mere parent existence as authority.
  6. Route every external persistent launcher through one CCS admission API/CLI. Retire public --print-id; replace it with a typed reserve-and-launch handoff that never turns a bare UUID into permission for arbitrary raw launch.
  7. Use protected local handoff storage or a Unix-socket broker for cmux/daemon launches. Never place claim tokens in command strings, process arguments, workspace configuration, or scrollback. Workspace restoration creates a fresh resume admission rather than replaying a consumed token.
  8. Install a stable CCS shim outside the updater-managed ~/.local/bin/claude symlink and point cmux’s Claude Binary Path (or PATH resolution) at it. Keep the generated identity aliases/functions: claude intentionally launches from $HOME, while claude-here preserves the current directory. The CCS shim must exec the dynamically resolved raw symlink target by absolute path, never call claude by name, and doctor-check the chain.
  9. At SessionStart, atomically claim valid external-launch reservations after Claude allocates the ID. Empirically verify continue:false behavior in interactive, print, and SDK modes; add UserPromptSubmit/PreToolUse denial as defense-in-depth for identities already marked violations.
  10. Model in-process succession separately. /resume activates an existing birth; /branch//fork create a new birth from an authenticated active-process capability; conversation recovery reactivates. Verify /clear ID behavior before assigning it to either path.
  11. For launcher/TUI forks, reserve before launch, prompt humans for user-managed versus supporting intent, and store forked_from_session_id separately. In-process fork intent needs its own default or later adoption rule because SessionStart cannot prompt.
  12. Implement native Agent admission across documented events: reserve in blockable PreToolUse by tool_use_id; record pending birth at SubagentStart by agent_id; bind through PostToolUse tool_response.agentId; reconcile async/background completion at SubagentStop.
  13. Require persistent Agent SDK hosts to integrate the CCS admission library/API and inject claim hooks/tokens programmatically or through managed policy. SDK persistSession:false remains exempt; persistent sessions that skip integration become visible violations.
  14. Keep API, raw gateway calls, --no-session-persistence, SDK persistSession:false, and CLAUDE_CODE_SKIP_PROMPT_HISTORY outside the local session catalogue.
  15. Persist reserved/claimed/active/exited/failed/expired/interrupted/unreconciled states per admission. Define which signal owns termination—shim wait, SessionEnd/Stop hook, or reconciler—and make reconciliation idempotent without deleting evidence or retrying work.
  16. Distinguish affirmative invalidity from CCS infrastructure failure. Invalid/replayed/mismatched claims stop; unavailable or corrupt claim infrastructure follows an explicit degraded-mode policy and writes a violation to a DB-independent append-only spool.
  17. Make human break glass shim-local and database-independent: interactive-only, mandatory reason, append-only evidence, explicit warning, and later reconciliation. Agents and automation cannot invoke it.
  18. Add human-only audited adopt and repair operations. Adoption changes effective management class without rewriting birth creator; repair appends corrections for parent/origin/bypass violations while preserving every prior value and reason.
  19. Make default visibility derive from effective management class, with creator, attestation, launch channel, admission state, and violation status independently filterable. Post-cutoff tokenless sessions remain visible unclassified violations.
  20. Roll out in explicit phases: schema and observe-only recording; shims/hooks plus doctor; migrate every cmux/daemon/launchd/script/SDK callsite and nested guard; warn-only cutoff; enforce; then remove legacy aliases after measured zero use.

What changed
