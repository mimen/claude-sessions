Model 02

Sessions are bodies. Identities are work units.

The migration must preserve this distinction instead of converting every historical label into durable structure.

Catalogue / session

Ephemeral body

A concrete conversation or runtime record. Legacy rows can remain useful here even when their durable identity is unknown.

Identity

Durable work unit

The stable thing that can outlive one session. Reconstruction requires authoritative configuration or explicit human mapping.

Grouping

Identity organizer

A structure that organizes identities. An old event value may inform one, but it is not automatically one.

Authority gap

No ~/.ccs-config role tree exists on this Mac.

Identity reconstruction therefore cannot be authoritative yet. The work laptop’s configuration is the missing source needed to resolve roles, identity keys, clusters, and role-derived loop behavior.

Collision policy

Current 19 rows win. On a matching session, merge only null fields from the legacy row. Never overwrite a non-null current value.

Recency policy

Never overwrite newer timestamps. Use updated_at as a gate; older legacy evidence cannot move current data backward.

Ambiguity policy

Unresolved rows remain unattached sessions. Their event, skill, and kind survive as legacy metadata pending a later decision.

Identity policy

No identity by resemblance. Similar titles or skill labels may generate a dry-run candidate, never an automatic durable work unit.

Route 06

Seven controlled stages, with apply held behind a human gate.

Each stage emits evidence for the next. No stage silently upgrades interpretation into fact.

  1. Snapshot & backupFreeze both databases and record hashes, versions, paths, counts, and timestamps.
  2. Inventory & dry-runRead both schemas and classify every legacy row without writing target data.
  3. Mapping registryDefine explicit role, event, identity, grouping, and legacy-meta rules.
  4. Disposable v34 copyRun the adapter against a copy of the current database—not the live file.
  5. ValidateCheck counts, referential integrity, collision outcomes, timestamps, and invariant totals.
  6. Human decision packetPresent ambiguous mappings, collision summaries, and unresolved identity questions.
  7. Transactional applyOnly after decisions; apply atomically, then perform post-apply verification.

Gates 07

Safety is expressed as invariants and stop conditions.

A migration is acceptable only when it can prove what it preserved, what it merged, and what it declined to infer.

Pre-apply invariants

  • Both source and target snapshots are readable and restorable.
  • The source inventory accounts for exactly 106 legacy rows.
  • The untouched target snapshot accounts for 19 catalogue rows, 0 identities, 0 groupings, and 0 inboxes.
  • Every legacy row has one dry-run disposition: create, null-only merge, unattached session, or rejected with reason.
  • No identity or grouping write lacks an explicit registry rule.

Immediate stop conditions

  • The adapter expects a column absent from the inspected schema.
  • A current non-null value would be overwritten.
  • A legacy timestamp would replace a newer current timestamp.
  • Identity, cluster, grouping, role, loop behavior, or resume_id would be guessed.
  • Any source row disappears from the reconciliation totals.
  • Validation changes the live database or runs outside the disposable copy.

Transactional apply gate

  • Begin one explicit transaction against a freshly backed-up v34 target.
  • Recheck target counts and timestamps immediately before writes.
  • Abort on any drift from the approved dry-run manifest.
  • Commit only after all post-write invariants pass inside the transaction.

Rollback posture

  • Rollback the transaction on any invariant failure.
  • If post-commit verification fails, restore the exact pre-apply target snapshot.
  • Retain the dry-run manifest and decision packet so the failure is reproducible.
  • Never use the legacy database itself as a mutable working copy.
