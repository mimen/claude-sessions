ccs × pr-watch · vision & design

Direct a fleet of agents,
not a handful of tabs

ccs maps disposable Claude Code sessions onto durable agent identities — each with a role, a lifecycle, a memory, and a mailbox. Making the management deterministic turns "how many agents can I keep track of" from a bottleneck into a non-issue.

Built on the cmux terminal · pr-watch is the first fleet.

▸ The problem

We're in the age of loops and multi-agent workflows. The way to get more done, faster, at higher quality is to run many agents at once — some you drive directly, some running autonomously. The work is there. The ergonomics are a mess.

Claude Code sessions have become Chrome tabs. They sprawl. You can't tell which ones matter. You lose the important ones in the noise. There's no lifecycle — no sense of a session being armed for a job, waiting on you, done, or safe to archive. A restart wipes the board.

The root problem: a Claude Code session is treated as disposable, but it's actually valuable — it holds real work and context. ccs flips that.

▸ Why warm agents, not subagents

Subagents are powerful, but they're fire-and-forget: dispatch, run, return, nothing warm afterward. That fits a bounded task. It does not fit ongoing work.

Real work is more like a project than a task: a pull request that gets review comments days later, an initiative that spans many turns. For those, a warm agent — persistent, holding its context, re-engageable when new feedback lands — is far preferable.

Subagent

Cold start. Bounded task. Returns once. No inbox. No lifecycle.

Good for: a lookup, a batch job.

Warm agent — what ccs builds

Persistent identity. Holds context across days. Has a mailbox. Re-engageable when new feedback arrives.

Good for: a PR from open to merge; an ongoing watcher; anything project-shaped.

▸ What ccs adds to Claude Code

ccs is three things at once: a TUI, a CLI of verbs, and a durable store. Its goal is to provide a deterministic harness around the non-deterministic behavior of agents — the agents can be creative; the machinery around them cannot.

A lifecycle for sessions

A session has states — armed, running, waiting on you, done, archived. You keep, resume, or discard deliberately, not by anxiety.

Deterministic roles

Each agent has a role with its own hooks and permissions. Agents of a role run the same way every time — that's what makes them trustworthy unattended.

Close & resume, losing nothing

Close any agent (or reboot) and reopen it exactly where it was — same role, same work, same messages.

A standard mailbox

A durable inbox per agent. One channel for agents to hand each other context and tasks — works even when the recipient is closed.

A legible terminal

Workspace tabs that bubble up what each session is — PR#, phase, epic — rendered from ccs so nothing drifts.

Fleets on autopilot

Group agents into a cluster and run them as one operation. Bring the whole thing back armed with a single command.

▸ The three pieces

ccs is the brain and the durable store. cmux is the terminal you actually look at. Claude Code is the runtime inside each session. Each piece does one job.

1 Identity — the primitive

ccs assigns an agent identity. A Claude Code session is the vessel that carries it. The identity is the durable thing — it persists whether or not any session is currently running it.

An identity's name is its responsibility — cluster · role · work-unit. For example, pr-watch · pr-agent · W-19338247. Everything else — the inbox, the state, the memory — keys off that.

durable survives everything

  • identity key — its stable name
  • mailbox — what others have told it
  • state — what it concluded

ephemeral comes and goes

  • current session — the Claude session id right now
  • body — the cmux tab, while open
  • cwd — the role directory it runs in
Why this matters Because the durable things don't depend on the ephemeral things, we can close an agent, re-open it under a new session, and it still has the same name, the same mailbox, the same memory. Nothing is keyed on the vessel.

2 Roles — first-class

A role is a first-class ccs entity. Defining a role once means every agent of that role is spawned, armed, resumed, and permissioned the same way — every time.

A role declares what it needs: its home directory, its permissions, its hooks, its skills and commands, and — for autonomous loops — how to arm itself on resume so it comes back running, not dormant.

The determinism payoff A role's hooks live in its directory. Every session of that role picks them up automatically. When you say "the control loop always drains its inbox and reads the board on start," that's not a convention — it's the hook that fires on every SessionStart. Behavior is the same every time, which is what makes it safe to run unattended.

3 Clusters — a fleet that works as one

A single agent is useful. The real power is a cluster: a group of agents that together run an entire operation. A cluster has a core (a few singleton roles that run the show) and a fleet (many workers of one role, one per unit of work).

pr-watch is the first cluster. It runs a team's whole PR pipeline — coordinates the epics that group the work, drives each open PR toward merge, runs agent code-reviews, answers reviewer feedback across GitHub and Slack — mostly on its own, pulling a person in only when a real decision is needed.

controlcore · loop
The driver. Reads the board every tick, routes the next task to each worker, unblocks stalls, flips gates as PRs become ready. Never talks to you.
conciergecore · loop
Your front door. Prioritizes everything in flight and brings you the most pressing thing — a decision, an approval, a review comment to answer. The only role that talks to you.
slack-scoutcore · loop
Reads Slack, classifies each message to a PR, drops it into that worker's inbox. Sensor only — never posts.
evalcore · loop
Grades how the loop is running and proposes concrete improvements.
pr-agentfleet · one per PR
The warm worker. Authors the PR, improves it, requests + acts on agent reviews, answers reviewer feedback — over days, staying warm the whole time. Identity keyed on the work item, not the PR#, so opening the PR never splits its mailbox.
designercore
Shapes the loops themselves. Writes ADRs and specs. Not in the live PR flow.

3 How the pr-watch roles talk

Sense runs on its own clock — a background loop that reads GitHub, CI, deploy, and Slack, then composes the board. Every role reads from the same board. The concierge is your default channel; you can steer control directly or drop into any worker.

4 Stages & the gate

Every PR moves through a fixed set of stages. The stage is sensed, not self-declared — the engine reads GitHub state and composes the stage. A worker cannot bump its own stage, and the machine is monotonic: no flaky signal can drag a PR backwards.

building→ milad-review→ in-review→ approved→ merged

A worker's job is to move its PR through the gate — a small set of checks that must all pass before it's handed to you. Control routes the worker each tick based on what the gate still needs:

worker-fix

CI red, conflicts, or reviewer changes requested. The worker knows what to do.

worker-assess

A reviewer left feedback that needs triage. The worker decides how to answer.

wait

CI running or an external review pending. Nobody owes work.

The Milad gate A PR clears agent reviews AND your own review before ANY public review is requested. That's why milad-review is a stage, not just a flag. Your /prwatch:approve command un-drafts the PR and lets CODEOWNERS auto-request review; you never have to un-draft by hand.

5 The board — one truth, many readers

Everything the fleet does resolves to the board — one composed view of every PR and every session, rebuilt every time a sensor ticks. It's the single source that the concierge, control, workers, and the TUI all read from.

concierge reads

Which PRs are at your review, which are approved-and-mergeable, which are in external review, which are merged and waiting to deploy.

control reads

Which alerts are firing and who owns them. Composes a task per alert, dispatches to the right worker's inbox.

workers read

Their own row at the top of every turn. Gate self-check — they don't wait to be told what's wrong.

Each row carries the stage, who it's awaiting (worker, you, reviewers, CI, deploy), and any firing alerts (CI red, changes requested, worker wedged, etc). Because everyone reads the same shape, the fleet never disagrees with itself.

6 State — everything durable, keyed by responsibility

An identity's durable state lives in ccs, keyed by its responsibility — so it outlives any one session. Close an agent, re-open it a week later under a fresh session, and it still has everything: its mail, its memory, the shared picture its cluster was working from.

what others tell me

Inbox. Every identity has one. One inbound stream of everything sent TO it — a task from control, a Slack message from the scout, a note from you. Durable, ordered, works while closed.

what I concluded

Identity state. The agent's own memory — its stage, its phase, what it decided, what it's working on next. Written by the agent as it works; rehydrates on resume so a fresh embodiment never starts cold.

the shared picture

Cluster state. The board, gate verdicts, dispositions, sensor timestamps. What the whole cluster reads from every tick. Written by sense; read by everyone.

All three key on the same address — the responsibility, not the session id. That's why continuity is of the identity, not of any one running process.

Deliver, then wake Loops notice new mail on their next tick. A worker doesn't loop, so a sender writes the mail then calls ccs bump-session. The wake is best-effort; the delivery is durable. If the wake fails, nothing is lost — the worker drains its inbox on its next start.

7 The display — the terminal, made legible

The point of ccs holding rich metadata isn't bookkeeping — it's ergonomics. ccs holds each session's meaningful facts (PR number, ticket, epic, phase) and projects them into three places you actually look.

The ccs TUI

The whole fleet at once, grouped by cluster and role. Open vs. closed, phase, dispatch state. Where you resume, archive, or tag sessions.

The cmux workspace tab

A phase pill + the session's real title, PR#, epic, W-number. Scan a window and read the whole fleet without opening a single one.

The Claude Code statusline

A clickable PR link + phase, always pinned in the pane. Same source as the tab — nothing to drift.

All three are projections of the same metadata, one-way. Store it once; it shows up wherever you look.

▸ A deterministic harness around non-deterministic agents

Agents are creative — their outputs vary run to run. The machinery around them can't be. ccs's job is to provide a deterministic harness so the non-determinism stays contained to the model turn itself, and everything surrounding it — routing, delivery, stage, dispatch — behaves the same way every time.

Sensed, not self-declared

Stage is composed from GitHub facts — draft, review decision, merge status. A worker can't set its own stage. There's no escape hatch.

One clock, per-sensor cadences

Every signal runs on a schedule you can inspect. The board is only as fresh as its slowest relevant sensor, and every value carries when it was written.

Delivery by identity, not by luck

A nudge routes to the right worker's tab via its identity → surface UUID. Never to "whatever window happens to be focused."

Fail loud, not silent

Every dispatch records its result. Un-confirmed deliveries surface as their own alert on the next tick. Nothing drops quietly.

▸ Why a cluster, not a dozen loose sessions

Without ccs, pr-watch is a dozen-plus Claude tabs you'd hand-manage and lose on restart. As a cluster:

One command back up

The whole operation returns armed. Every loop is ticking. Every worker knows what it owns.

Control re-tasks the fleet

The board recomposes. Each worker's next task lands in its inbox by the end of the first tick.

Overnight Slack is already routed

The scout's drops from the last 12 hours are already in the right worker mailboxes when you sit down.

Concierge surfaces only what needs you

The few things at your review or blocked. Everything else is running without you.

Any worker is still one click away when you want to dive in. The fleet runs autonomously and stays open to you.
