1. Why we're doing this

Today catalogue is a fat table where every session carries all the identity-relevant columns (role, cluster, pr_repo, pr_number, gus_work, work_unit_id, grouping_id, stage, status_line, meta) directly. "Identity" is a derived string (key) that deriveKey(row) re-computes on every mutation — a cache of a computed value, not a foreign key. Twin sessions for the same PR are two rows that happen to share identity columns; there's no entity that is "the pr-agent responsible for owner/repo#12345."

Because sessions are their identity, every consequence of that confusion has to be papered over: deriveKey drift between TS + Python (fixed once, could drift again), twin-detection logic in the board composer, ccs stage/ccs mark/ccs status writing per-session what should be per-identity, and a CLI surface that grew by accretion because there was no unifying entity to hang verbs on.

The refactor makes identity a real entity. A session points at an identity. Identities carry the durable attributes (role, cluster, grouping, work-ref, stage, status_line). Sessions carry only per-run state (title, resume_id, parent, meta scratch). Two claude sessions working the same PR share one identity row. The key cache dies; identity_key becomes a foreign key.

Concurrent wins this unlocks:

  • The CLI collapses ~60% because ccs identity set <key> --grouping=… --pr_state=… replaces ~10 per-session command names.
  • Onboarding a new fleet role type = drop a folder with role.toml + identity-schema.toml. Ccs materializes the storage automatically.
  • Inboxes + state + dispositions collapse from filesystem primitives into SQL tables — one transactionality story, no more mkdir-locks.
  • Stage/status_line correctness across respawns: a new spawn inherits its identity's fields instead of re-sensing from scratch and briefly showing stale values.
  • Roll-ups ("how many identities per grouping", "how many pending inbox items per role") become one SQL query instead of ad-hoc file scans.

The pr-watch cluster stays exactly one concrete cluster. The refactor's real payoff is the shape: any future cluster (issue-watch, sprint-tracker, whatever) inherits the model, ships as a folder, and works with zero ccs changes. That's the shareability story turned inside-out — clusters compose against a clean platform instead of tunneling their vocabulary into ccs.

Core versus fleet roles

Every role declares its kind in role.toml: fleet (many identities per cluster, one per work item) or core (exactly one identity per cluster, the tuple (cluster, role) IS the identity).

KindIdentity countStorageExample
core1 per clusterOne identities row per (cluster,role)concierge, control, designer, scout, eval
fleetN per clusterOne identities row per work item + a matching per-role table rowpr-agent, review-agent

The distinction matters because only fleet roles need a per-role attributes table. A concierge has no PR number, no epic; its state is just (cluster, role) + stage + status_line — all universal columns. A pr-agent has PR facts, GUS facts, screenshots — the per-role table holds those.

Session ↔ identity link

Sessions carry identity_key as a soft foreign key. A session with identity_key = NULL is a "loose" session — the designer scratch buffer, a one-off transcript, anything that doesn't fit a role. Loose sessions are still indexed and browsable; they just don't participate in the identity join.

Every mutation to a session's cluster/role reuses or mints the correct identity. ccs new-session --identity <key> becomes the primary spawn primitive; if the identity doesn't exist and the caller has role + cluster + work-ref, it's minted lazily.

Identity_key format

Structured, human-readable, deterministic:

<cluster>:<role>:<work_ref>

# fleet
pr-watch:pr-agent:owner/repo#12345
pr-watch:review-agent:owner/repo#12345

# core (work_ref is elided — the tuple is the identity)
pr-watch:concierge
pr-watch:control

The tool never parses these — they're opaque to ccs. But being structured makes them greppable, URL-safe, and easy to reason about in prompts and logs.

3. Database schema

The migration bumps catalogue.db from v31 to v32.

CREATE TABLE identities (
  identity_key      TEXT PRIMARY KEY,     -- 'pr-watch:pr-agent:owner/repo#12345'
  cluster           TEXT NOT NULL,
  role              TEXT NOT NULL,
  kind              TEXT NOT NULL,        -- 'fleet' | 'core'
  grouping_id       TEXT,                 -- FK -> groupings.grouping_id

  status_line       TEXT,                 -- worker's freeform pill

  updated_at        TEXT
);
CREATE INDEX idx_identities_cluster ON identities(cluster);
CREATE INDEX idx_identities_role ON identities(role);
CREATE INDEX idx_identities_grouping ON identities(grouping_id);

CREATE TABLE groupings (
  grouping_id       TEXT PRIMARY KEY,     -- e.g. 'a3QEE000002SAsv2AG'
  cluster           TEXT NOT NULL,
  role              TEXT NOT NULL,        -- which fleet role uses this grouping
  label             TEXT,                 -- 'Team Tokens UI'
  url               TEXT,                 -- deep link
  short_name        TEXT,                 -- column-friendly
  notes             TEXT,                 -- JSON array of strings
  context           TEXT,                 -- long-form authored context
  closed            INTEGER NOT NULL DEFAULT 0,
  meta              TEXT,
  updated_at        TEXT
);
CREATE INDEX idx_groupings_cluster ON groupings(cluster);
CREATE INDEX idx_groupings_role ON groupings(role);

CREATE TABLE inboxes (
  inbox_id          INTEGER PRIMARY KEY AUTOINCREMENT,
  identity_key      TEXT NOT NULL,        -- FK -> identities.identity_key
  from_role         TEXT,
  message           TEXT NOT NULL,
  status            TEXT NOT NULL DEFAULT 'pending',  -- 'pending' | 'drained'
  created_at        TEXT NOT NULL,
  drained_at        TEXT
);
CREATE INDEX idx_inboxes_identity_status ON inboxes(identity_key, status);

CREATE TABLE identity_state (
  identity_key      TEXT NOT NULL,        -- FK -> identities.identity_key
  key               TEXT NOT NULL,
  value             TEXT NOT NULL,        -- JSON

  PRIMARY KEY (identity_key, key)
);

CREATE TABLE dispositions (
  disposition_id    INTEGER PRIMARY KEY AUTOINCREMENT,
  cluster           TEXT NOT NULL,
  subject_key       TEXT NOT NULL,        -- what got decided (e.g. an identity_key or a work-ref)
  verdict           TEXT NOT NULL,        -- 'approve' | 'hold' | 'defer' | 'clear'
  reason            TEXT,
  decided_by        TEXT,
  decided_at        TEXT NOT NULL,
  meta              TEXT
);
CREATE INDEX idx_dispositions_cluster_subject ON dispositions(cluster, subject_key);

4. Role config: role.toml + identity-schema.toml

Every role's folder gains role.toml. Fleet roles additionally have identity-schema.toml declaring their per-role table columns.

role.toml (universal)

# ~/.ccs-config/clusters/pr-watch/roles/pr-agent/role.toml
kind = "fleet"                # 'fleet' | 'core'

[grouping]
type = "epic"                 # display label for groupings of this role's work
mint_policy = "external"      # 'external' = ids from tracker; 'internal' = ccs mints
url_pattern = "https://gus.lightning.force.com/lightning/r/ADM_Work__c/{grouping_id}/view"
# ~/.ccs-config/clusters/pr-watch/roles/concierge/role.toml
kind = "core"                 # no [grouping], no per-role table, no identity_shape
Where's stage? Nowhere in role config. The identities.stage column is an opaque string as far as ccs is concerned — ccs stores it and returns it. The engine (e.g. catalogue_sync.py) owns the vocabulary, computes stage from its sensor inputs every tick, applies whatever ordering rule the cluster wants (pr-agent uses a monotonic clamp against a rank table), and writes through ccs identity set --stage=…. Ccs never reads role.toml to validate a stage write — it just stores whatever string it gets. This keeps cluster vocabulary out of the platform: a rogue write is a data race that the next sensor tick corrects, not a schema violation. Same treatment for anything else that felt like it needed platform validation.

identity-schema.toml (fleet only)

# ~/.ccs-config/clusters/pr-watch/roles/pr-agent/identity-schema.toml
# Declares the per-role identity table (identity_pr_agent).
# Ccs materializes matching columns on boot; additive-only.

[columns]
pr_repo         = { type = "text", indexed = false }
pr_number       = { type = "integer", indexed = true }
pr_state        = { type = "text" }
pr_head_sha     = { type = "text" }
pr_branch       = { type = "text" }
gus_work        = { type = "text", indexed = true }
gus_work_sf_id  = { type = "text" }

# composite index for fast PR lookup
[indexes]
pr_lookup = ["pr_repo", "pr_number"]

[display]
primary   = ["pr_repo", "pr_number"]      # what shows in "#12345 owner/repo" columns
subtitle  = ["pr_state"]                  # dim secondary text
short     = "pr_number"                   # narrow-column pick

Type vocabulary is a fixed set: text, integer, real, boolean. Anything else is a validation error at boot.

Change rules:

  • Add a column — additive; ccs runs ALTER TABLE ADD COLUMN on next boot.
  • Add an index — additive; CREATE INDEX IF NOT EXISTS.
  • Rename a column — role authors a migration note under [migrations] in the schema; ccs runs ALTER TABLE RENAME COLUMN once per note (tracked by hash). Explicit, not inferred from diff.
  • Drop a column — same as rename: explicit [migrations] entry.
  • Change a column's type — not supported; author a new column, migrate values in role code, drop the old.

5. Per-role table materialization

At ccs boot (or when a role's schema changes), ccs walks every cluster in ~/.ccs-config/clusters/*, reads each fleet role's identity-schema.toml, and reconciles the DB:

  1. Compute the target schema for identity_<role_slug> (dashes→underscores).
  2. Read the current DB schema for that table.
  3. Diff:
    • Missing table → CREATE TABLE.
    • Missing columns → ALTER TABLE ADD COLUMN.
    • Missing indexes → CREATE INDEX IF NOT EXISTS.
    • Migration entries in [migrations] whose hash hasn't been applied → run each, record hash in schema_migrations table.
  4. Fail loud if a column's declared type doesn't match the DB's existing type.

Materialization is idempotent; running it twice does nothing on the second pass. Failure aborts boot with a clear message. The reconciler code is ~150 LOC + tests.

Guardrail: ccs never drops columns implicitly. A schema that removes a column silently would silently delete production state. All destructive changes go through [migrations] — the role owner has to declare intent.

10. Migration script

One migration bumps schema v31 → v32. It's a single transaction; if any step fails the entire migration rolls back and the tool refuses to boot until the state is investigated.

Phase 1 — schema

  1. Create identities, groupings, inboxes, identity_state, dispositions, schema_migrations.
  2. Read every cluster's role folder; materialize per-role tables (identity_pr_agent, etc.).

Phase 2 — data migration

  1. For each row in catalogue:
    • Compute identity_key via today's deriveKey() (unchanged logic).
    • If the row has enough info to identify a role + cluster + work-ref, UPSERT into identities.
    • If the row's role is fleet, UPSERT into the matching identity_<role> table with PR/GUS/work_unit_id columns.
    • Update catalogue.identity_key to the derived key.
  2. Load ~/.ccs/clusters/*/cluster/groupings.json; for each entry, INSERT into groupings.
  3. Walk ~/.ccs/identities/*/*/*/inbox/*.json; INSERT each message into inboxes.
  4. Walk ~/.ccs/state/**/*.json; INSERT each key/value into identity_state.
  5. Load ~/.ccs/clusters/*/cluster/dispositions.json; INSERT into dispositions.

Phase 3 — sessions table

  1. Rename catalogue → sessions.
  2. Drop the columns now on identities: role, cluster, kind, project, event, pr_repo, pr_number, pr_branch, pr_state, pr_head_sha, gus_work, work_unit_id, grouping_id, key, stage, status_line, resume_command, completed, archived, parked_task_id.
  3. SQLite drops columns via ALTER TABLE DROP COLUMN (3.35+); fallback to table rebuild if not available.

Phase 4 — file cleanup

  1. Move each cluster's inbox files to {cluster}/inbox.migrated/ (rename, don't delete — recoverable).
  2. Same for state/, groupings.json, dispositions.json, sessions.json, pending.json, work-units.json.
  3. User can delete .migrated directories manually once satisfied.

Rollback

The migration keeps a catalogue.db.v31.bak copy before touching anything. If ccs boots on the migrated DB and detects a corrupted or half-migrated state, it refuses to boot and points at the backup. Rolling back to v31 is: replace catalogue.db with the backup, restore the .migrated dirs.

Warning: the migration is one-way. Going back to a pre-refactor ccs binary against a v32 DB will fail — the columns don't exist. Keep the backup until you're confident the new shape is stable.

11. Engine + hooks + skills rewrite

Every subprocess call site + prose reference in the pr-watch cluster updates in the same release. Rough counts:

Engine subprocess call sites

~88 ccs invocations in engine/scripts/*.{py,sh}

Concentrated in: spawn-agent.sh, catalogue_sync.py, approve.py, retire_sweep.py, mark_completed.py, gate_eval.py, compose_board.py, membership.py.

Hook + skill prose refs

~156 ccs command mentions across roles/*/.ccs-hooks/*.md and .claude/skills/**/prompts/*.md

Concentrated in: pr-agent stop-context, concierge SKILL.md, control SKILL.md, reviewer skills.

ccs_client.py — the single Python indirection

Today wraps ccs catalogue export, ccs identity resolve, ccs session-fields. Post-refactor:

catalogue_export(cluster, role=None)          # unchanged; still emits flat JSON envelope
identity_resolve(session_id)                  # unchanged
identity_set(identity_key, fields, sensor=None)  # replaces session_fields for durable attrs
session_set(session_id, fields)               # per-run attrs only
inbox_send(identity_key, from_role, message)
inbox_drain(identity_key)
grouping_upsert(grouping_id, cluster, role, label, url, ...)

Each engine script rewires its calls. catalogue_sync.py is the biggest: currently emits a ccs session-fields call per PR with a blob of fields; splits into an identity_set for durable fields (pr_repo, pr_number, grouping_id, gus_work) and a session_set for per-run (customTitle, stage — actually stage lifts to identity so it drops from session_set).

spawn-agent.sh — the spawn path

The ensure command mints an identity if it doesn't exist, then spawns a session pinned to it:

# Before:
ccs new-session --cluster pr-watch --role pr-agent --key owner_repo-12345 \
  --pr-number 12345 --pr-repo owner/repo --title "…" --cwd "$wt"

# After:
key="pr-watch:pr-agent:owner/repo#12345"
ccs identity mint "$key" --cluster=pr-watch --role=pr-agent 2>/dev/null || true
ccs identity set "$key" --pr_repo=owner/repo --pr_number=12345
ccs session new --identity="$key" --title="…" --cwd="$wt"

Hook + skill search-and-replace

Systematic 1:1 replacement per the deletion table. Prose examples like "run ccs mark $CCS_SESSION_ID --completed" become "run ccs identity complete $CCS_IDENTITY_KEY" — which means hooks also need to know the identity_key of the current session. That's a new env var (CCS_IDENTITY_KEY) exported by the SessionStart hook, derived from ccs identity resolve --session $CCS_SESSION_ID.

13. Risks + mitigations

Risk 1 — migration lands broken on the live fleet

The pr-watch cluster runs 24/7 with real PRs. A broken migration takes the fleet offline until fixed.

Mitigation: migration test with a real v31 fixture pinned in the repo. Explicit backup file (catalogue.db.v31.bak) created before any writes. Migration runs behind an env flag (CCS_ALLOW_MIGRATE_V32=1) on first run so the operator opts in. Rollback documented in the migration script's docstring.

Risk 2 — per-role table materialization has subtle bugs

Type coercion, index creation ordering, column-add-then-index race conditions in SQLite.

Mitigation: ~30 unit tests on the materializer alone. Boot-time reconciliation fails loud on any mismatch. Fixed type vocabulary (text/integer/real/boolean) — no user-defined SQL.

Risk 3 — prose search-and-replace misses a reference

~156 mentions in hooks/skills. A missed one means a worker runs a nonexistent command mid-loop.

Mitigation: a post-refactor lint script that greps every .md under ~/.ccs-config for the deleted command names and fails CI. Runs as part of the mutation-boundary test.

Risk 4 — engine writes an unknown role's data and materialization hasn't run

A new fleet role is declared in role.toml, but ccs hasn't booted since the change. Engine tries identity_set against a role whose table doesn't exist yet.

Mitigation: identity_set runs materialization on-demand if the target table is missing. Materialization is idempotent + fast, so the extra check is free.

Risk 5 — TUI natural-language commands emit stale command names

Codex/claude have been trained on prompts that mention the old CLI. They'll emit ccs mark $id --completed instead of ccs identity complete.

Mitigation: the NL command prompt lists the current CLI explicitly, not from memory. Update the system prompt to match.

Risk 6 — no gradual rollout means "big bang" debugging

Everything ships at once. If something's wrong, you can't isolate which piece.

Mitigation: land in a feature branch, run the full test suite + a full pr-watch end-to-end simulation before merging. Merge on a low-activity day (weekend). Have the rollback plan rehearsed.

Order chosen so each step is verifiable independently and later steps can't proceed without earlier ones landing.

  2. Schema v32 migration + universal tables. Create identities, groupings, inboxes, identity_state, dispositions, schema_migrations.   3. Per-role table materialization. Reader for role.toml + identity-schema.toml. Reconciler that runs on boot. Tests. Materialize identity_pr_agent, identity_review_agent from pr-watch's role folders.
  4. Grouping CLI + migration. Move groupings.json → groupings table. ccs grouping commands. Tests.
  5. Inbox CLI + migration. Same shape for inbox files → inboxes table. ccs inbox commands.

  8. Cluster + hook CLI reshape. ccs cluster board/catch-up/decide/resume/…, ccs hook run/explain/lint. Tests.

16. pr-watch reimagined — skills, hooks, sense scripts

The best test of whether this refactor is worth doing: pick each surface (skills, hooks, sense scripts), look at the mental gymnastics it does today because sessions carry identity, and show how it collapses when identity is first-class. Skills are prose that gets easier to write; hooks are shell that gets shorter; sense scripts are Python that stops re-deriving identity on every tick. All three simplify — sense scripts and hooks the most, because they do the mechanical work.

Concierge — read the queue

Today (SKILL.md, 315 lines). The skill spends 40+ lines explaining how ccs board pr-watch --json joins "catalogue + poll + review + github_review_state + gate + worker_activity" and emits "one row per IDENTITY (cluster:role:work-unit — the durable axis, unaffected by session churn/twins)." Then more lines on why truth.audience is legacy but still emitted, on how data.stage vs meta.milad_review interact, on which "reserved lane" concierge can write (dispositions).

The reader has to know: what the board composer is, that sessions get de-duped into identities, that some fields are legacy, that some state files (pending.json, convo.json, review_relay.json) are still authoritative for their sub-domains, that concierge can only write dispositions.

Post-refactor. "Read the queue" is one command:

ccs identity list --cluster=pr-watch --role=pr-agent --stage=milad-review

Returns identity rows, each already carrying stage, status_line, grouping, pr_repo, pr_number, live-sessions count, last-updated. No "board composer" as a concept the skill has to explain — the board is the identity table, filterable directly. No legacy fields, no twin de-dup logic, no reserved lanes. The skill's queue-reading section drops from ~50 lines to ~5.

Every filter recipe collapses similarly:

  • Milad's queue → --stage=milad-review
  • Ready-to-merge → --stage=approved
  • External review sitting → --stage=in-review
  • Merged, deploy pending → --stage=merged

Concierge — surface a decision

Today. Milad approves #12345. The skill runs /prwatch:approve, which internally: resolves the current session's identity, updates meta.milad_review, invokes compose_board.py synchronously to force a fresh recompose, so the sidebar doesn't paint stale. The command wraps ~80 lines of Python.

Post-refactor. The identity IS the durable thing. Milad's approval writes to it directly:

ccs identity set pr-watch:pr-agent:owner/repo#12345 --meta.milad_review=approved

The board composer's job stays the same, but the identity row now reflects the write immediately — no "resolve session → identity → write" round-trip. The /prwatch:approve command shrinks to a one-liner. Sidebar reads live identity state; freshness is automatic.

pr-agent (worker) — drain your inbox

Today (pr-agent.md, 341 lines). The first mechanical step of every task:

ccs inbox drain --cluster pr-watch --role pr-agent --work-unit <your-key>

The worker has to know its "work-unit" (a repo_pr-flat-key like owner_repo-12345), plus the cluster + role slug, to construct the addressing tuple. The subagent-briefing.md stamps this key into every dispatched task so workers don't have to compute it, but the concept still exists in the skill and every drain call has three flags.

Post-refactor. The worker has one identity — its identity_key is in $CCS_IDENTITY_KEY, exported by the SessionStart hook. Drain is:

ccs inbox drain $CCS_IDENTITY_KEY

No cluster+role+work-unit tuple to remember or embed. Inboxes are indexed by identity_key directly — one flag, unambiguous. The briefing stamp for the inbox key can be deleted. The worker prompt gets shorter and less error-prone.

pr-agent — record a verdict

Today. The worker writes a verdict into ~/.ccs/clusters/pr-watch/cluster/pr-owner_repo-12345.judgment.json. The file path is convention, not enforced. The spawn.json hook precreates it as {} so the worker's first write is an edit (not a create — which would prompt in acceptEdits mode). Fragile.

Post-refactor. Every identity has a canonical scratch directory:

$(ccs identity path $CCS_IDENTITY_KEY)/judgment.json

Which resolves deterministically to ~/.ccs/clusters/pr-watch/identities/pr-agent/owner_repo-12345/judgment.json. The precreation lives in the identity's path --new flag, called once at spawn. No convention to memorize; no risk of workers writing to a slightly-wrong path and getting orphaned files.

Control — dispatch a task

Today (spawn-agent.sh, ~320 lines). Spawning a pr-agent for PR #12345 involves: computing the flat key owner_repo-12345, finding the W-number by parsing board.json, minting a session id via ccs new-session with 7 flags (--cluster, --role, --key, --title, --pr-number, --pr-repo, --gus-work, --cwd), spawning a cmux workspace to run claude --session-id $sid, then upserting the sessions.json registry with 3 fields atomically. The complexity is because identity is computed from session-scoped fields, and every spawn has to re-derive it.

Post-refactor. Identity is minted (or reused) first, session is pinned to it:

key="pr-watch:pr-agent:owner/repo#12345"

# Mint the identity if new (idempotent — no-op if exists)
ccs identity mint "$key" --cluster=pr-watch --role=pr-agent

# Set the PR attrs (or update them — engine does this every tick anyway)
ccs identity set "$key" --pr_repo=owner/repo --pr_number=12345

# Spawn a session pinned to it
ccs session new --identity="$key" --cwd="$wt"

No W-number lookup (the identity carries it, or doesn't — either way not needed at spawn). No sessions.json registry (dispatch state moves to identity_state). No atomic multi-field session write (identity set does it). spawn-agent.sh loses ~120 lines.

Control — retire a worker

Today (retire_sweep.py, ~140 lines). Walks every non-retired pr-agent session in the catalogue (via ccs catalogue export), fetches fresh PR merge state from GitHub for each, applies the is_complete predicate, and calls ccs mark $sid --completed to retire the session. A PR that spans multiple sessions (twins, respawns) gets marked per-session — retiring one twin doesn't retire the identity, so the sweep has to iterate.

Post-refactor. Retirement is per-identity, not per-session:

for key in $(ccs identity list --cluster=pr-watch --role=pr-agent --json | jq -r '.[].identity_key'); do
  merge=$(gh pr view ...)   # engine's fresh-merge-state fetch
  if is_complete "$merge"; then
    ccs identity complete "$key"    # one call retires ALL sessions for this identity
  fi
done

The sweep iterates identities, not sessions. Retiring an identity cascades: its sessions all lifecycle-transition, its inbox stops accepting new messages, the board drops the row. Concept collapse: "worker retired" becomes a single fact, not N per-session facts.

catalogue_sync — the hot-path composer

Today (~350 lines). Every tick, for each pr-agent in the roster: read the full catalogue via ccs catalogue export, look up the session by identity key (built from pr_repo+pr_number), compute the stage, compute the epic id, build a payload of 5 fields (customTitle, cluster, stage, gusWork, groupingId, meta.gus_work_sf_id), call ccs session-fields $sid --json '...' --sensor=catalogue-sync. Repeats for every session, including twins.

Post-refactor. The sync writes to the identity, not the session. Twins share the write. Fewer subprocess calls:

ccs identity set "$key" --stage=milad-review --grouping=$grouping_id \
                        --pr_state=OPEN --pr_head_sha=$sha --sensor=catalogue-sync

One call per identity (not per session). The customTitle stays a session-level attr — separate ccs session set if we want per-tab-name updates. And stage flows through the same call as PR attrs — no separate stage-write ceremony. The composer's per-tick loop drops ~30 lines and roughly halves its subprocess count.

Board rendering (compose_board.py)

Today (~700 lines). Composes the per-worker truth view by joining: catalogue rows (via ccs catalogue export) × gate.json × convo.json × sessions.json × groupings.json. Twins have to be de-duplicated by identity_key. The output writes to board.json keyed by identity, not session — the composer already discovered this abstraction, it just didn't have a name.

Post-refactor. The composer starts with identities and joins outward:

SELECT i.identity_key, i.stage, i.status_line, i.grouping_id,
       p.pr_repo, p.pr_number, p.pr_state,
       g.label AS grouping_label, g.short_name,
       (SELECT COUNT(*) FROM sessions s WHERE s.identity_key = i.identity_key) AS session_count
FROM identities i
LEFT JOIN identity_pr_agent p USING (identity_key)
LEFT JOIN groupings g ON g.grouping_id = i.grouping_id
WHERE i.cluster = 'pr-watch' AND i.role = 'pr-agent';

One query, no twin logic (identities are unique by definition), no state-file joins for groupings (now a table). The gate.json + convo.json joins still exist — those are per-tick sensor outputs — but the twin/identity plumbing goes away. Composer shrinks by ~150 lines.

Hooks — where the mechanical work lives

Hooks are the least-glamorous surface but the one that touches every session on every event. Every start/stop/spawn/paint fires them; a fragile hook is a fleet-wide bug. Post-refactor they get materially simpler because they no longer have to compute identity from session-scoped fields.

pr-agent stop-context (126 lines today)

Today. Runs at the end of every worker turn to keep the tab pill current. Contains one particularly gnarly bit:

jq -r --arg key "$(ccs meta . --json 2>/dev/null | jq -r '.prRepo + "#" + (.prNumber|tostring)')" \
  '.data.prs[$key] | {route, pending: …}' \
  ~/.ccs/clusters/pr-watch/cluster/gate.json

What that does: read this session's catalogue meta, extract prRepo + prNumber, build a flat key, use it to look up the row in gate.json. Three concept hops (session → catalogue → key → gate row) to answer "what does the gate say about my PR?"

Also has a long section explaining "you do not set stage or activity" — because the old model made ccs stage writable, and every prompt tuning cycle had to remind workers to stop pushing stage manually.

Post-refactor. The worker's identity_key is in $CCS_IDENTITY_KEY at hook time. Gate lookup collapses to:

ccs identity $CCS_IDENTITY_KEY --json | jq '.gate'

The board composer emits gate.route as an attribute on the identity row; no gate.json shell join needed. The whole "you do not set stage" paragraph deletes — workers can't set stage cleanly because there's no per-session stage anymore; stage is on the identity and the sensor owns it.

Stop-context shrinks from 126 lines to ~70. What's left is the actual content the hook is FOR: shortname/status-line guidance, novel-context rules, forbidden phrases. All the plumbing goes away.

SessionStart hook (across all roles)

Today. When a session starts, the hook has to figure out the session's identity to set up env vars, tab pills, statuslines. It does this by reading the catalogue row for the current session id and pulling role/cluster/prRepo/prNumber/gusWork/workUnitId out. Every role's start hook has this same lookup boilerplate.

Post-refactor. The session row has identity_key as an FK; the hook resolves once:

export CCS_IDENTITY_KEY=$(ccs identity resolve --session $CCS_SESSION_ID)
export CCS_CLUSTER=$(echo "$CCS_IDENTITY_KEY" | cut -d: -f1)
export CCS_ROLE=$(echo "$CCS_IDENTITY_KEY" | cut -d: -f2)

Every hook downstream (statusline, cmux-paint, spawn) uses those env vars. The lookup boilerplate collapses to one line per hook. And because identity_key is stable across session churn (respawns share it), env-var caching works properly for the first time.

spawn.json (pr-agent) — the perm-grant paths

Today. The spawn hook grants Bash + Write perms scoped to specific paths — Write({home}/.ccs/clusters/pr-watch/cluster/**), Edit({home}/.ccs/clusters/pr-watch/cluster/**). But the per-worker judgment/result files live in that whole directory, so the grant is broader than needed — every pr-agent can write every other pr-agent's judgment file. Not a security issue in this trust model, but sloppy.

Post-refactor. Each identity has a dedicated scratch dir. The perm grant scopes exactly:

"allow": [
  "Write({home}/.ccs/clusters/pr-watch/identities/{role}/{identity_slug}/**)",
  "Edit({home}/.ccs/clusters/pr-watch/identities/{role}/{identity_slug}/**)",
  "Bash(ccs identity {identity_key}:*)",
  …
]

The hook templating already supports variables like {cwd}; adding {identity_slug} is trivial. Now each worker's file-write blast radius is its own directory. Same shell surface, tighter scope.

meta-update hook (pr-agent)

Today. Fires when the catalogue writes a meta field on this session (e.g. sensor updates meta.milad_review). Uses this to update the cmux tab pill. But it fires per-session, so twin sessions each fire independently — and if they have different meta snapshots (race between two sensor writes), the tab flickers.

Post-refactor. The meta is on the identity, not the session. Twins share one row; sensor writes update it once. The hook fires per-identity-change, not per-session-write. Flicker disappears; the hook simplifies from "diff my session's meta against last-seen" to "read my identity's current meta."

Sense scripts — the biggest wins

The 14 pr-watch sense scripts are ~8,100 lines total. Every one of them today has to bridge the "sessions carry identity" model to the "I want to know about the PR" reality. Post-refactor that bridge disappears — sensors read + write identities directly.

catalogue_sync.py (341 lines)

Covered above. The per-PR write loop iterates once per identity (not per session), calls one ccs identity set instead of a stapled session-fields blob. Twin sessions no longer produce duplicate writes.

compose_board.py (818 lines — the biggest single script)

gate_eval.py (503 lines)

for pr in open_prs:

New capabilities that were awkward before

Beyond simplification, the model unlocks things pr-watch skills couldn't do cleanly today:

"How many PRs per epic are in milad-review?"

SELECT g.label, COUNT(*) FROM identities i
JOIN groupings g USING (grouping_id)
WHERE i.stage = 'milad-review'
GROUP BY g.grouping_id;

Today: parse board.json, group by grouping_id in Python, join to groupings.json. Now: one query. The concierge can render epic-level rollups in the render step without ad-hoc code.

"Which identities have unread messages older than an hour?"

SELECT i.identity_key, COUNT(inb.inbox_id) AS unread
FROM identities i
JOIN inboxes inb ON inb.identity_key = i.identity_key
WHERE inb.status = 'pending' AND inb.created_at < datetime('now', '-1 hour')
GROUP BY i.identity_key
HAVING unread > 0;

Today: walk the inbox filesystem, correlate with catalogue rows. Now: one query. Control can catch stuck-inbox scenarios trivially.

"Show me every identity in this epic across all its sessions"

An epic view in the TUI or CLI:

ccs grouping a3QEE000002SAsv2AG --identities

Renders every pr-agent working on Team Tokens, their stages, live sessions, last activity. This view didn't have a home in the old model — the closest was ccs cluster pr-watch, which showed everything undifferentiated. Now it's a first-class query.

"Compare two clusters running the same shape"

If pr-watch is joined by an issue-watch or ticket-watch cluster, cross-cluster queries land trivially because identity + inbox + state are shared universal tables:

SELECT cluster, stage, COUNT(*) FROM identities
WHERE role IN ('pr-agent', 'issue-agent')
GROUP BY cluster, stage;

Today: not possible without ad-hoc scripting. Post-refactor: one query. This is what "the platform stays generic, clusters compose" actually buys.

The pattern behind all the wins

Every simplification above is one of three shapes:

  1. Eliminating "session → identity" indirection. Today: "look up this session's PR, find its identity by matching PR columns, act on it." Post-refactor: identity is right there, act directly.
  2. Replacing filesystem addressing with FK addressing. Inboxes, dispositions, scratch dirs — instead of "compute the path from role+cluster+key," it's "query by identity_key." One-flag calls, no path drift.
  3. Making rollups first-class. Group-by-epic, group-by-cluster, per-identity aggregates all become SQL queries because the data lives in joinable tables instead of parallel JSON files.

None of these are new features. They're all things pr-watch already does, just done cleanly instead of via workarounds. The refactor's value shows up as "the workaround stops being needed" every time you touch one of these files.
