1. Executive summary

Both vendors independently reached the same overall verdict: the core model is genuinely strong and durable. The catalogue/index split, the responsibility→identity→embodiment→lineage model, the three-homes split (tool / git-tracked config / machine-local runtime), the identity-keyed board contract (ADR-0077), durable worker inboxes, the control/concierge plane split, and deterministic sensing feeding an inference layer are all called out as correct, deep design by both reviews. Sol: "a good core model wrapped in too much machinery." Fable: "the risks are not in the core architecture."

The problems cluster in four places:

  1. The branch is mid-migration broken. The ADR-0077 board rewrite landed the composer but not its consumers: catalogue sync fails every tick and gate results never reach the board. Verified directly in code, not just by the reviewers.
  2. Two hard blockers for any second cluster: role resolution is global-by-name (not cluster-scoped), and the identity key has no single source of truth (3–4 independent implementations across TS and Python, with drift already logged internally).
  3. State integrity is convention, not mechanism: unlocked read-modify-write on sessions.json and board.json, ownerless lock recovery, hollow watchdog liveness, singleton guarantees removed without replacement.
  4. The real tool↔cluster interface is the private SQLite schema, because the Python engine reads ccs databases directly. Every tool migration is secretly a two-repo, two-language migration.

None of these require rethinking the design. They define Phase 1: a hardening pass on the work laptop that leaves pr-watch provably correct and the platform seams enforced, after which this machine's ccs-config build-out starts from a solid contract.

2. Part I: Bug review, all findings

Sources: Fable reviewer-A (code-traced), GPT-5.6 Sol reviewer-B (code-traced), orchestrator verification. CONFIRMED = the reviewer traced the failing path in code. PLAUSIBLE = strongly inferred, not fully traced. Old-layer findings already tracked in the repo's own REVIEW.md and PRODUCTION-READINESS-CHECKLIST.md are not repeated here; see Appendix.

P0: broken now or blocks everything

B1. Two incompatible board/gate contracts are live in the same pipeline

P0CONFIRMEDSolverified by orchestrator ccs-config engine

compose_board.py (main(), ~line 599–620) now writes the ADR-0077 shape: top-level {status, provenance, rows, today, sprints, senseHealth, counts, prs (legacy), ticketedNoPr}. But catalogue_sync.py:163–169 still requires the legacy nested shape: it checks "board" in board and reads board["board"]["prs"]. It therefore fails with "board.json missing" on every tick and returns before synchronizing stage, grouping, title, GUS metadata, or PR facts to the catalogue.

Second split in the same pipeline: compose_board.py:559 reads gate_eval.json, but gate_eval.py:492 writes gate.json. Nothing writes gate_eval.json, so gate results (satisfiesTicket, scope-drift, visual-review-stale alerts) never reach the composed board. catalogue_sync.py:170–175 also still reads gate.json.

Why it persists silently: sense-scheduler.py:91–100, 203–211 runs catalogue_sync every tick and logs failures without halting. The board, catalogue, and rendered pills can disagree indefinitely while everything reports healthy. This is exactly the "engine-sensor desync" failure class the cluster.toml version gate was written to prevent; it slipped through because the seam is inside the engine where no gate exists.

Fix shape: one board shape, one gate artifact name, and an end-to-end tick test (sensor → gate → composer → catalogue_sync → rendered row) run against fixtures in CI.

B2. Role resolution is global by name, not (cluster, role): hard blocker for any second cluster

P0CONFIRMEDSol ccs tool

resolveRole(role) scans cluster directories and returns the first match by role name (src/roles/role-files.ts:123–134). Used unscoped by: resume (resume-session.ts:95–99), cluster resume/pinOnResume (resume-cluster.ts:181–191), new-session cluster inference / work-unit semantics / cwd / resume command (new-session.ts:154–169, 228–251), stage validation (catalogue/commands.ts:244–253), and the resume-command cache keyed by role name alone (catalogue/db.ts:445–467).

Failure scenario: both pr-watch and a second cluster will have roles named control, concierge, scout, eval. Whichever cluster sorts first owns those names machine-wide: resuming event-watch:control can execute pr-watch's /pr-watch-control loop in pr-watch's cwd and validate against pr-watch's stage vocabulary.

Related: ensure-control.sh:42–72 detects control liveness by searching cmux for the literal workspace title "CONTROL PLANE": a second cluster reusing the title false-positives both clusters' watchdogs. Also src/tui/clusterView.ts:22 hardcodes legacy system-name fallbacks (pr-watch-control, pr-watch-2...) that could misclassify a second cluster's sessions (Fable; low risk, delete).

Fix shape: role identity is (cluster, role) everywhere including caches, hooks, colors, spawn, resume, pinning; watchdog discovery keys on ccs identity, never workspace title. Design note: Milad's future direction includes standalone/floating roles, so build the fix so cluster is a namespace, not a hard requirement baked deeper than it must be (a floating role can be modeled as a cluster-of-one).

B3. Identity key has no single source of truth (3–4 implementations, drift already logged)

P0CONFIRMEDFable both repos

The cluster:role:work-unit key, the keystone concept of the whole platform, is independently implemented in:

  • src/catalogue/lineage.ts:23 identityKey(): priority workUnitId > pr:/gus: derived > role:
  • src/catalogue/db.ts:545–548 identityKeyOf(): reads the stored row.key column, which is populated only by explicit setKey() calls (new-session.ts:140, catalogue/commands.ts:160), never auto-derived
  • compose_board.py:128–136 identity_key(): a hand-rolled Python reimplementation of the same priority order
  • The repo's own docs/FINDINGS-dry-determinism.md:7 records "work-unit key logic implemented 6x, 2 drifted", rated P0 internally, still unfixed on this branch

Failure scenario: when the Python-derived board identity and the TS-derived catalogue.key disagree for the same work unit, src/board/indexer.ts:59–73 bySession() falls back to identityKeyOf(catRow); a mismatch silently returns null: the board row exists but the tool reports no board data. Every new cluster adds another reimplementation.

Fix shape: ccs derives and stores the key on every mutation and exposes it via CLI/JSON export; engines read it and never re-derive. (Subsumed by B10's export boundary.)

P1: state integrity, singletons, watchdogs

B4. Control watchdog can report a dead control plane as healthy forever

P1CONFIRMEDSol ccs-config engine

Health = "a titled workspace exists AND board.json is fresh" (ensure-control.sh:30–36, 50–70). But board.json is written by the independent launchd sensor pipeline (sense-scheduler.py:194–211; the pill-sweeper cadence alone makes composer runs near-continuous, :70–80). A wedged/dormant Claude control session keeps its workspace while launchd keeps the board fresh: the anti-dormancy watchdog reports "alive + ticking" while the system is permanently dormant. Additionally, when it does detect a titled-but-stale control it logs and refuses to recover, deliberately leaving the system wedged (Sol design review).

Fix shape: a control-owned monotonic heartbeat token (written only by the control session's own tick), watchdog logic = alive+fresh → no-op; dead → ccs resume the MRU control embodiment; alive-but-stale → mark degraded + one operator action. Recovery must go through ccs resume control --cluster pr-watch, never a raw cmux/claude command (current raw launch at ensure-control.sh:74–88 bypasses identity, manifest, and hook contracts).

B5. Unlocked check-then-spawn: duplicate control planes

P1CONFIRMEDSol ccs-config engine

ensure-control.sh:59–81: check and spawn are separate unlocked operations. Two overlapping invocations (launchd + manual, or clock skew) can both observe no control workspace and both call cmux new-workspace. No lock, CAS, lease, or post-create reconciliation. Two control loops then dispatch workers, drain events, and mutate catalogue state from different snapshots. Compounds with B7.

B6. One-embodiment guard removed without an atomic replacement (docs still promise it)

P1CONFIRMEDSol ccs tool

src/catalogue/spawn-contract.ts:80–109 tolerates duplicate embodiments; docs/CONTEXT.md:81–88, 140–150 still promises one embodiment per work unit; sibling superseding is best-effort and decoupled from process launch (new-session.ts:167–196). Two concurrent spawns can both pass validation and launch; superseding doesn't stop already-started processes; atomic inbox drain doesn't protect a shared worktree from concurrent edits/commits/pushes.

Nuance (Sol's design review, worth honoring): MRU-preference over a hard invariant is arguably right for a single operator; the dangerous cases are same-session double-resume and the control singleton. Decision D6 below: pick which layer gets a hard atomic claim (control: yes; workers: preference + loud warning may be enough). Whatever the choice, docs and code must say the same thing.

B7. sessions.json: unlocked read-modify-write, multi-step partial states

P1CONFIRMEDSol ccs-config engine

spawn-agent.sh:42–50 registry_set rewrites the whole registry per field; a worker registration calls it three times (sessionId, cmuxWorkspace, cwd; :137–157). No lock: concurrent worker starts clobber each other; a crash mid-sequence leaves a valid-looking but incomplete record (sessionId with no cwd/workspace) → hooks stop recognizing the worker, dispatch concludes no session exists. migrate_keys.py:83–105 is worse: mutates several entries and renames state files before persistence, then does a direct non-atomic overwrite; interruption detaches result/judgment files from registry identity.

B8. Stale sense-lock recovery can delete a newer run's active lock

P1CONFIRMEDSol ccs-config engine

sense-run.sh:22–25, 41–56: one shared lockdir; any lock older than 30 minutes is removed on mtime alone; every owner unconditionally removes the same path on exit. Sequence: run A exceeds 30m → B deletes A's "stale" dir and recreates → A exits and deletes B's active lock → C starts concurrently with B. No owner token, PID validation, or ownership-safe cleanup. This also makes the fixed temp paths in sense_common.py:42–56 and sense-scheduler.py:118–122 unsafe under exactly the condition the lock exists to contain.

B9. Single-row board recompose is unlocked (dormant today, lost-update bug the day it's wired)

P1CONFIRMED (dormant)Fable ccs-config engine

compose_board.py:628–659 compose_single() does read-prior → merge one row → atomic write, without taking sense-run.sh's lock. Atomic rename protects readers from torn files, not writers from lost updates: an event-triggered single-row recompose racing a scheduled whole-board compose can silently revert every other row to the pre-tick snapshot. No callers exist yet (the ccs board --identity/--recompose tool-side PR is step 5 of the migration spec), so this must be fixed in the same PR that wires the callers, and called out in that PR's acceptance criteria.

B10. The engine reads ccs's private SQLite directly; the schema is the real interface

P1CONFIRMEDSolboth design reviews both repos

compose_board.py:39–42, 89–149 queries catalogue columns and reimplements identity precedence; worker_activity.py:75–95 attaches both databases. This violates the platform's own "neither reaches into the other" rule: the compatibility interface is de facto the DB schema, not cluster.toml or the board contract, so every schema migration is a coordinated two-repo, two-language migration.

Fix shape (both vendors, same answer): stable machine-readable projections from the tool: ccs cluster <name> --json, ccs catalogue export --cluster <name> --json, ccs identity resolve --session <id> --json (the existing src/catalogue/cluster-map.ts:240–281 projection is close; deepen it). Forbid direct DB reads from clusters. This single change also retires B3's drift class at the root.

P2: contract enforcement, hooks, migrations, liveness edge cases

B11. requires_ccs is barely enforced; board.json is never runtime-validated

P2CONFIRMEDSolFable ccs tool

Only resume-cluster calls checkClusterGate (cli.ts:615–633); new-session spawns without gating (new-session.ts:228–251); board compose/read paths parse the manifest but skip the gate (recompose.ts:51–76, board-command.ts:104–167); missing/malformed manifests and same-major shortfalls warn-and-continue (manifest.ts:134–174, 154–160), which is weak for a pre-1.0 tool where 0.1 vs 0.9 is a real gap.

Separately (Fable): src/board/paths.ts:27–38 readBoard() is JSON.parse + a bare as Board cast, while the manifest layer got a real zod schema (manifest.ts:57–70). A malformed row from a second, less-dogfooded composer parses fine and then crashes deep in buildMaps() (indexer.ts:23–33) or the TUI/cmux paint code with no "board malformed" signal at the boundary.

Fix shape: zod-validate board.json on read against a versioned schema; run the cluster gate on every bring-online path (new-session, board commands, resume).

B12. activity was dropped from storage/CLI (migration v30) but hooks and docs still depend on it

P2CONFIRMEDSol both repos

db.ts:427–435 drops the activity column; no CLI command remains; but the worker Stop hook still calls ccs activity ... and swallows every failure (worker-stop-hook.sh:45–74). A blocked worker's needs-you escalation now silently no-ops: the worker-owned escalation edge is gone, not just doc lag. docs/CONTEXT.md:105–114, 169–187, phase-state-machine.md:1–56, and runbook.md:45–66 still teach stage x activity as current.

Related design footgun (Fable): as designed, activity had dual write-authority (engine senses "fixing", worker self-reports "needs-you") in one field with no precedence rule enforced at the write boundary. When re-introducing the escalation edge, either split sensed vs self-reported fields or enforce precedence in the ccs command.

B13. Worker stop-hook events: non-atomic write, one mutable filename per key

P2CONFIRMEDSol ccs-config engine

worker-stop-hook.sh:35–43 writes event JSON directly to the final path events/<key>.json: crash mid-write leaves malformed JSON; multiple turns before a drain overwrite each other. No sequence number, turn ID, journal, or temp+rename. If events are meant to be the return half of each worker turn (as comments state), the semantics are weaker than documented; if they are only a level-triggered "something changed" hint, document that and keep it.

B14. Contradictory-but-parseable liveness sources can resolve wrong

P2PLAUSIBLESol ccs tool

src/cmux/bridge.ts:160–168 acknowledges activeSessionsBySurface can be stale after reattachment, yet inserts that view first and refuses to replace an existing surface binding (:175–203). If the active map binds a live surface to an old session while sessions[currentId].surfaceId says otherwise, the stale value wins and the live-tree intersection can't resolve it (the surface itself is valid). ADR-0054's fail-closed guard covers unreadable sources, not contradictory-but-valid ones. Potential effect: resume skips the dead identity and relaunches the live one, or makes the wrong duplicate decision.

B15. Hand-patched live DB can be silently blessed as current by presence-driven migrations

P2PLAUSIBLESol ccs tool

v16–v19 migration guards check only column-name existence (db.ts:273–311); later renames/drops are also presence-driven (:362–435); the DB is stamped v30 with no post-migration schema validation (:436). Because the live DB was hand-patched during v16–v19, a column with the wrong type/default, or both old and new names present, can cause a migration block to skip its operation and still stamp current (e.g. if both system and cluster exist, v27 skips the copy and still records success). Fix shape: a startup schema postcondition validating required columns, forbidden legacy combinations, and data-transfer invariants before advancing user_version; plus migration smoke tests (fresh + fixture-of-live-shape).

B16. Operational papercuts (Fable)

P2CONFIRMEDFable
  • launchd plists are copy-paste with hardcoded absolute paths and labels (engine/deploy/com.mimen.pr-watch-*.plist); no generator. Every new cluster hand-authors 2–3 plists. Small templating script recommended.
  • worker_activity.py wedged-detection (git log + inbox mtimes off catalogue.cwd) was not fully traced by any seat; audit for timezone/off-by-one before reusing the "worker went quiet" pattern in another cluster.
  • Engine test coverage: 2 test files under engine/lib/ vs 30+ scripts under engine/scripts/, against near-total test coverage in the TS tool. Sensing is where truth drifts; it is the least-tested layer.
Sol's design smell, worth keeping verbatim: the deepest issue is not code volume but partially overlapping control surfaces that can disagree: catalogue stage vs composed board phase; sense.sh vs sense-scheduler.py; gate.json vs gate_eval.json; catalogue identity vs sessions.json routing identity; ccs lifecycle enforcement vs shell-level cmux checks; platform hooks vs legacy global hooks; documented stage x activity vs the v30 stage-only implementation. Atomic helpers, a manifest gate, a liveness bridge, and a board contract all exist, yet writers and entrypoints routinely bypass each one. Phase 1's real job is concentrating invariants at single boundaries.

3.1 Tool/cluster boundaries

Agreed strong: the ADR-0061 litmus test ("would a second cluster inherit it?") is honored in the TS core: stage/activity are blessed columns with cluster-declared values, work units and groupings are generic entities with cluster-typed display, board/types.ts keeps workUnit.kind and owner open strings. Three homes (tool / git config / runtime) map cleanly to change cadence and rebuild safety. Self-contained cluster packages with package-relative engine paths are right.

Agreed weak: the compatibility contract is a version range with no schema; the engine invocation is a shell-out whose real interface is "whatever fields pr-watch's Python happens to read/write"; and B10 (direct SQLite access) means the schema is the true contract. Sol's concrete proposals, endorsed:

  • Versioned executable protocol in cluster.toml: declare argv + protocol number + timeout for board and sense entries instead of assuming python3 <file>.
  • Versioned board schema validated at publish and read.
  • Runtime state classification: ~/.ccs currently conflates "not git-tracked" with "disposable". Split into durable operational state (inboxes, processed-message history, dispositions, judgments/results, watermarks) vs rebuildable derived state (index, boards, sensor snapshots, pills, caches). This drives backup, reset, migration, and repair-by-deletion, and it becomes the seed of the multi-machine state-home answer later.
  • Board recompose semantics: synchronous single-row recompose is becoming an orchestration protocol attached to every mutation (catalogue/commands.ts:175–298). Prefer: writes append an identity-scoped changed event; the composer coalesces; one bounded synchronous refresh operation exists for interactive commands; consumers display lastComposed honestly.

3.2 Maintainability

Agreed: maintainable if a consolidation phase happens now. ~26k lines TS + ~8k Python/shell is fine for one person + agents. The entropy hot spots, in order:

  1. ADRs as chronological memory. 78 ADRs with live contradictions: two ADR-0001 files; the ADR-0053 numbering collision (already known); CONTEXT.md says 8 hook types, registry has 10 (hook-types.ts:39–80); ADR-0032 (one-embodiment invariant) vs ADR-0073 (preference); ADR-0060 keeps catalogue stage while ADR-0077 authorizes deleting it; the config repo's CONTEXT.md is still written in pre-rename vocabulary (system/constellation/event, retired paths); the scout SKILL cites the retired ~/.claude-sessions/catalogue.db path. Agents amplify doc drift: each session can find a locally plausible stale doc and perpetuate it. Fix: one curated current-architecture document; every ADR gets a status (active / superseded-by / partially-active / abandoned / migration-only); regenerate or delete stale prose docs (the tool repo's GLOSSARY/SYSTEM-UNITS "extracted from ground truth" pattern is the right one; apply it to cluster docs or delete them).
  2. Prompts becoming programs. The control skill (230 lines) embeds a full dispatch algorithm, ownership matrix, and recovery policy as prose; the concierge skill (300+ lines) embeds board rendering, suppression bookkeeping, relay mechanics, and merge policy. Prose policy accumulates exceptions faster than code because appending a paragraph is free. See 3.6 for the extraction list.
  3. Cross-language duplication (identity, paths, state envelopes): fixed by B10's export boundary + a tiny state SDK.
  4. Two repos: correct split; it is expensive only because the seam is porous. Fix the protocol, not the repo count.
  5. Polyglot: TS tool + Python engine is fine. Shell should shrink to launchd wrappers and env setup: sense-run.sh is appropriately thin; ensure-control.sh implements liveness/recovery policy in shell and talks to cmux directly, which belongs behind ccs or in Python.
  6. Engine tests: bring the sensing layer up to the tool's test discipline (fixtures per sensor, golden-file composer tests, the e2e tick).

3.4 UX

Agreed hierarchy, each surface one job: cmux pills = ambient glance (two-pill budget praised by both) · TUI = navigate/focus/resume · board = complete operational truth · concierge = interpretation + decisions · CLI primitives = agent/machine interface.

  • TUI is several products at once (catalogue, cluster console, transcript reader, skill browser, cost dashboard, metadata editor, NL command line, titler, lifecycle editor; six views, many unlabeled single-key ops). Make cluster / recent / search the three primary modes; demote the rest behind a view picker; move mutations behind a command palette.
  • CLI namespace flat-mixes operator verbs and internals. Stable operator surface: ccs, ccs cluster <name>, ccs board <name>, ccs resume <selector>, ccs new, ccs inbox, ccs doctor. Everything else documented as the machine interface. Business verbs (ready/approve) stay cluster commands, not platform verbs.
  • Help text under-represents ~a third of real commands (role, resume-command, gus-work, epic, phase; many flags). Cheap, high-value fix given the surface IS one person's muscle memory.
  • Add ccs why <session>: plain-language dump of identity, lineage, stage source, hook resolution, board row. Today debugging requires holding the full 15-term ontology; this one command absorbs most of that load.
  • Concierge should not re-dump five tables per digest. Unsolicited contact only for time-sensitive/blocking items; concise delta on re-entry; full board only when asked. (Matches the chosen hybrid speaking-rights model: funnel initiates, tabs converse.)
  • Four channels render the same state (pill, TUI, concierge, status line) with no built-in cross-check; drift reads as the system lying from multiple mouths. The board-as-single-source fix (D5) plus a periodic consistency assertion in the e2e test covers this.

3.6 Deterministic vs inference boundary

Agreed principle, agreed gap: "scripts prepare, agents decide" is right and sensing is correctly deterministic; the failure direction is deterministic coordination living in prompts as model-executed pseudocode. Concrete moves (Sol's list, endorsed, with Fable's additions):

  1. Dispatch loop → plan_dispatch.py. The control skill's algorithm (filter awaiting-worker rows, in-flight dedup, alert→task mapping, inbox write, dispatch bookkeeping) is already deterministic. Script emits {dispatches, escalations, ambiguous}; control reasons only over ambiguous + cross-PR conflicts.
  2. Alert→task templates as data (ci-red → fix-and-push, conflicts → rebase), not prose the model re-executes faithfully.
  3. Concierge rendering → prwatch operator-view --format json; the model decides what to mention and when, never re-formats rows.
  4. Dispositions → prwatch decide <item> <disposition>: records suppression, writes inbox/control events, refreshes state atomically. The current suppression recipe in the concierge prompt is implementation leakage.
  5. Activity derived from observables (commits, CI, inbox writes, state writes) via worker_activity; keep only needs-decision-class self-reports as inference; enforce write precedence at the command boundary (see B12).
  6. "Ready for review" stays inferential but gate-verified: the worker proposes, the deterministic gate confirms before milad-review publishes.
  7. Moves the other way (mechanism → inference): almost nothing. The one candidate is ambiguous chat classification with confidence thresholds and candidate fallback: precision over coverage, exactly as the scout skill already states. This is also the critical pattern for event-watch (see Part III).

4. Part III: Event-watch port wargame

Seat: wargamer (Fable). Full report preserved in the session; key content in full below since it drives Phases 2–4.

4.1 Hidden assumptions that break the port

  1. Per-machine platform vs multi-machine fleet. ADR-0041 scopes everything (catalogue, runtime, cmux socket, launchd with hardcoded HOME) to one machine; launchd does not fire during sleep; there is no multi-host story. Resolved by steering decisions: single machine (standing computer) first; per-session placement later; state-home question explicitly deferred. Design tax now: keep identity keys, inbox addressing, and the board contract machine-neutral; the catalogue's per-host tagging becomes load-bearing.
  2. The event-slug anchor is unbuilt. ADR-0030 name-checks event-watch but ADR-0069 ships four anchor types (pr / gus / freeform / none) and freeform has no auto-reconnect across sessions. A real event-slug anchor with reconnect is a ccs tool change, not config.
  3. Monotonicity is load-bearing in three places (role.toml declaration, catalogue_sync max-latch, composer STAGE_RANK) and no non-monotonic cluster has ever run; tool-side stage enforcement is queued, not landed. Event lifecycle is genuinely non-monotonic (postponements reopen promo). Mitigant: the skeleton (announce date, event date, settlement window) is clock/Airtable-deterministic; a mostly-deterministic stage with a small non-monotonic escape hatch is achievable.
  4. "Workers never speak" is pr-watch's constitution, not the platform's, but it is load-bearing: board routing uses a PR-shaped awaitingFrom multiset; single-writer-per-field is the only safety on mergeFields. Event-watch has N+2 writers (coordinator, scout, every worker) and workers converse. Redesign the write topology, don't copy it. Steering decision (hybrid speaking rights) is compatible: workers converse but never initiate; flags go on the board; the concierge/assistant tier owns initiation.
  5. The tool still contains pr-watch literally: CORE_ROLES hardcoded (ADR-0062 queued), phase-rubric gates on role=="pr-agent" (ADR-0063 queued), ready/approve are PR verbs in the platform command layer.
  6. Name collisions with the live system: skills named event-watch / event-scout / event-worker / event-eval already exist and are running; install.sh symlinks by bare name into shared ~/.claude/skills/ and wires global settings.json hooks (where SessionStart/Stop double-registration is already a known P0). Side-by-side shadow migration requires renaming the new cluster's roles (e.g. ew-scout). The "CONTROL PLANE" title collision is B2's sibling.

4.2 Deterministic engine over LLM-classified inputs: where the pattern breaks and what holds

  • Confident-wrong beats fail-loud: the FAILED-status safety net catches crashes, not hallucinations; a misclassified Granola note is a well-formed fact the engine will latch.
  • The monotonic latch amplifies misclassification: one wrong "settle" becomes permanent via max(sensed, stored) and monotonic=true rejects the correction.
  • Reclassification churn produces ghost deltas and re-surfaced items.
  • Cadence economics: API polls are free; LLM classification per tick is not, and Slack reads already rate-limit.
The port rule that holds (both the wargamer and Sol's design review converge on it): LLM output may only write candidates and routed artifacts, never stages or lifecycle. The engine senses only deterministic facts (Airtable event records, dates, Shotgun sales, folder mtimes, session liveness); low-confidence classifications go into a candidates array the coordinator surfaces. The sanctioned maturation path is the eval skill's own doctrine: make every judgment yourself, then promote a stable pattern into a sense-helper.

4.3 Cross-cluster interference risks

  • Shared catalogue = blast radius: one SQLite DB for all clusters, migrations untested (B15); a botched schema bump takes down every cluster on the machine. Catalogue backup + migration smoke tests are a Phase 1 item for this reason, not a nicety.
  • Resume runaway doubles: the 2026-07-10 duplicate-fleet incident is why agents never resume the cluster; two clusters double the liveness-read surface. ADR-0054 fail-closed + B4/B5 fixes are the mitigation.
  • launchd/cmux contention: two pill-sweepers at short cadence on one socket; label namespacing per cluster required (B16 templating).
  • Dual supervision: loop-manager already re-arms dead loops fleet-wide; ensure-control does the same per-cluster. Teach loop-manager a "ccs-managed: hands off" exclusion the same day any cluster goes live on this machine.

4.4 What the current skill-based event-watch does better (what a naive port loses)

  1. Adaptive pacing: /loop + ScheduleWakeup ticks minutes-fast on event day, hourly in the lull; ccs assumes cron cadence. Resolution: keep model pacing inside sessions, launchd ensure-alive as backstop (control is itself a /loop under a cron guardian, so the platform already permits this).
  2. Multi-machine span: current fleet survives laptop sleep via the Mini; platform cannot express this yet (deferred by decision).
  3. Direct worker↔Milad channel: each event tab is a live conversation surface. Preserved by the hybrid speaking-rights model.
  4. Iteration speed: skills are markdown edited live; versioned cluster config is the right discipline once semantics are stable, drag while they are fluid.
  5. No tool↔engine seam to break (pr-watch's docs call that seam "the risk surface").

In exchange the platform buys: real anti-dormancy, born-correct spawn, durable exactly-once inboxes, identity-keyed boards, supersede-dedup: the things loop-manager currently approximates by hand.

4.5 Wargamer's ranked risks

  1. Machine topology mismatch (now resolved by decision; residual: keep addressing machine-neutral)
  2. Shared-catalogue blast radius (untested migrations + resume-runaway now endanger other live loops)
  3. LLM output entering the deterministic core (quarantine rule above; monotonic=false path unbuilt)
  4. Live-system name collisions (skills, hooks, workspace titles)
  5. Tool changes required, not optional (CORE_ROLES, phase-rubric, event-slug anchor, awaitingFrom vocabulary)
  6. Dual supervision (loop-manager vs ensure-control)
  7. No sensed ground truth for "done" (settlement is a judgment; needs coordinator-proposed / Milad-confirmed completion policy)
  8. Multi-writer state without locks (fixed in Phase 1; event-watch must still design its write topology)
