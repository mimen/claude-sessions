TL;DR + verdict

Where we stand: the architecture is production-grade. Every P0 from the previous full-system review (identity-key drift, role-name shadowing, broken sense pipeline, wedged control watchdog) is closed with tests + ADRs. Module boundaries in the tool are clean — the tool inventory audit scores it A− overall. The pr-watch cluster runs the whole PR pipeline reliably.

Where we're NOT: shareable. Moving to a second machine today means:

  1. Run-from-source Bun script with a symlink to a working tree. No build, no versioned artifact, no ccs update. Ships whatever's committed on feat/…, not a numbered release.
  2. ADR-0078 half-done in the cluster: we moved 2 Python scripts onto ccs catalogue export, but 6 more still import sqlite3 and open catalogue.db directly. Exactly the schema-is-the-interface coupling the ADR promised to kill.
  3. CLI namespace is flat — ~40 subcommands mix operator verbs, machine APIs, and internal plumbing. New users can't tell which surface is stable.
  4. TUI has direct DB mutations that bypass the CLI's command layer (writes catalogue via setCustomTitle, setCompleted, setArchived directly), risking drift as invariants get added.
  5. No new-cluster wizard. Every new cluster is hand-scaffolded from the toy-second template.
  6. Hardcoded absolute paths in every LaunchAgent plist (/Users/mimen/…), no bootstrap script, no README that reflects reality on both repos.

What "shareable" would look like:

  • Build + version story. ccs ships as either (a) an npm-published Bun-runnable package that bunx can run, OR (b) a compiled single binary via bun build --compile. Version in package.json gets bumped on each release, git tag matches, ccs --version matches.
  • ADR-0078 finished. Every Python script in the cluster reads the catalogue via ccs catalogue export. Zero import sqlite3 in cluster code.
  • CLI reorganized under nouns. ccs cluster …, ccs role …, ccs board …, ccs api …, ccs session …. The flat namespace becomes a short list of operator verbs (resume, new, ls, tree) at top level.
  • TUI mutations flow through the CLI's command layer. Or the CLI's commands become thin wrappers around a shared mutation module the TUI also uses. Same invariants either way.
  • New-cluster wizard. ccs cluster init <name> scaffolds cluster.toml, CHANGELOG.md, one starter role, and (optionally) a LaunchAgent plist templated with $HOME.
  • Portable LaunchAgent installer. A script that sed-replaces /Users/mimen with $HOME and runs launchctl bootstrap.

The good news: the architectural work is done. The remaining work is packaging, namespace hygiene, and finish work on ADR-0078 — not new design. Realistically 2–4 focused days of work.

Problems (both audits flagged)

  • Flat namespace mixes audiences. An agent-facing ccs catalogue export lives next to an operator-facing ccs mark next to internal-only ccs hook run. New users can't tell what's stable.
  • Help text under-represents the surface. The 96-line inline help lists ~30 commands; the actual switch covers ~40. role, gus-work, epic, several flags are missing.

Proposed reorg (both audits converge)

  • Root (top-level operator verbs): ccs (TUI), ccs resume <selector>, ccs new, ccs ls, ccs tree, ccs reindex.
  • ccs cluster …: ls, map, resume <c>, board <c>, gate <c>, init <c> (wizard).
  • ccs role …: ls, upsert, rm, sync.
  • ccs session … (or keep at root): rename, mark, tag, stage, meta, etc.
  • ccs api … (machine surface, self-documenting): catalogue export, identity resolve, cluster --json, board get.
  • Keep hidden: hook run, register-session, self-check, statusline, backfill-work-units. Document as machine interface, not user surface.

The redundancy question (your framing)

You asked: does the TUI reimplement what the CLI already offers? Answer: partially.

  • Data reads are shared. TUI + CLI both call getAll(), lifecycleOf(), identityKeyOf(), boardIndex(), allGroupingsAcrossClusters(). Both route resume through resumeSessionEntry — one shared core.
  • Formatting is duplicated. SessionList.tsx (236 lines) paints rows for the TUI; ccs ls (cli.ts:375–419) has its own formatter. Cluster view is rendered by renderClusterMap (CLI) vs src/tui/clusterView.ts (TUI). Independent presenters.
  • Mutations partly bypass the CLI's command layer. Cursor's audit called this out specifically: App.tsx:199–214 writes titles via setCustomTitle(catalogue, sid, title, now); App.tsx:580–588 and :697–704 toggle lifecycle bits via setCompleted / setArchived. Meanwhile the CLI's commands.ts adds validation + recomposeForSession(id) after mutations. The TUI would miss those side effects if they became load-bearing.
Recommendation: extract a shared mutation module that both the CLI's commands.ts and the TUI call. The db.ts setters stay as low-level primitives; a new mutations.ts wraps them with validation + recompose triggers + logging. Both surfaces call the wrapped versions. This is the correct answer to "which side is source of truth" — neither. The mutation invariants are the source of truth.

Catalogue layer A boundary · A testability

src/catalogue/ — 18 files, ~4.2k lines. Owns the durable session metadata SQLite (~/.ccs/cache/catalogue.db).

Key files

  • db.ts (1010 lines) — schema, 31 migrations (v1–v31 additive-only), all setters. Sole writer of catalogue.db.
  • commands.ts (561 lines) — CLI-facing command implementations. Validate → call setter → recompose → print result.
  • command.ts (184 lines) — TUI-facing mutation dispatcher. Same setters, different UX.
  • cluster-map.ts (282 lines) — roster (core + fleet) rendering + JSON projection.
  • board-command.ts, export-command.ts, identity-command.ts — the D1 read boundary from ADR-0078.
  • sync-tabs.ts, render-tab.ts, render-statusline.ts — cmux tab painting.
  • disposition.ts, lineage.ts, resolve-work-unit.ts, spawn-contract.ts, spawn-location.ts, stage-schema.ts — focused helpers.

Boundary quality

Excellent. Only db.ts touches the sqlite handle. Every setter stamps updated_at. Migrations are additive-only with a startup postcondition (ADR-0081 / B15). Identity key auto-derived on every mutation (ADR-0078). The mutation-boundary rule (ADR-0068) is enforced by convention — commands never do raw SQL.

Concurrency note

Each setter is a single UPDATE; no transaction wrapper. SQLite WAL + 5s busy_timeout mitigate, but two concurrent ccs invocations writing the same session's meta could race. In practice this only matters if we ever run multiple concurrent mutating writers on the same session — currently we don't (the sensor writes; workers read).

Shareability — How ccs runs today

This is the section that answers your "I don't even know how it works" question.

The invocation chain

$ which ccs
/Users/mimen/.bun/bin/ccs

$ ls -la /Users/mimen/.bun/bin/ccs
lrwxr-xr-x  ...  ccs -> ../install/global/node_modules/claude-sessions/bin/ccs

$ cat /Users/mimen/.bun/install/global/node_modules/claude-sessions/bin/ccs
#!/usr/bin/env bun
import { main } from "../src/cli.ts";
process.exit(await main(process.argv));

What this means, in plain English

  1. Your ccs command is a symlink into the Bun global-install area.
  2. That target is another symlink to ~/projects/claude-sessions/bin/ccs.
  3. That file is a 4-line Bun script that imports ../src/cli.ts and calls main().
  4. Bun runs the TypeScript directly. No compilation, no build, no dist/, no versioned artifact.

Every time you invoke ccs, Bun reads your live source files. If you edit src/cli.ts, the very next ccs invocation reflects the change. There is no "install" that captures a version — the tool is whatever's on disk right now.

How the symlink got created

package.json declares "bin": {"ccs": "bin/ccs"}. Running bun link from the project directory installs the symlink. That's what bun run setup (scripts/setup.ts) does — plus optional dep checks for claude/codex/cmux.

Version story (or lack of one)

WhatState
package.json "version""0.1.0" — never bumped
Git tagszero — git tag --list returns nothing
ccs --versionPrints 0.1.0 — same as package.json
Cluster gateReads package.json version, compared against cluster's requires_ccs
ChangelogPer-cluster CHANGELOG.md exists (ADR-0058). Tool has no CHANGELOG.
Release processNone documented.

The infrastructure for versioning exists — cluster.toml can declare requires_ccs = ">=0.2.0", and checkClusterGate enforces it. But the input to that gate (pkg.version) never changes, so the mechanism is inert.

Distribution paths — what could work

  1. Publish to npm. npm publish whenever we bump. Consumers bunx claude-sessions or npm install -g. Still requires Bun for bun:sqlite imports.
  2. Bun compile. bun build --compile src/cli.ts --outfile ccs produces a single-file executable. Distribute via GitHub Releases per platform. No Bun required at runtime.
  3. Homebrew tap. Wraps option 2 with brew install mimen/ccs/ccs.

My recommendation: option 2 (Bun compile) for the actual binary + option 3 (Homebrew) for the distribution channel. Bun compile is a single command, produces a native binary, and there's no npm-registry maintenance overhead.

~/.ccs-config portability

What's portable

  • cluster.toml / role.toml / CHANGELOG.md — pure config, no absolute paths.
  • Skills (SKILL.md), commands, hooks — all use relative refs.
  • Python scripts — use Path.home() / ".ccs" / ... (portable).

What's NOT portable

Every LaunchAgent plist in clusters/pr-watch/engine/deploy/ hardcodes /Users/mimen:
  • <string>/Users/mimen/.ccs-config/…/sense-run.sh</string>
  • <key>HOME</key><string>/Users/mimen</string>
  • Log paths, working directories, all absolute.
On a machine where $HOME is /Users/alice, launchd will fire the plist against a path that doesn't exist. Silent no-op.

Fix

Two options:

  1. Install-time templating: ship the plists as .plist.template files with __HOME__ placeholders. An install.sh substitutes at install time.
  2. Runtime resolution via launchctl setenv: put $HOME in the plist, install a shim script that resolves. More complex, less portable.

Option 1 is standard practice. ~30 lines of shell for the templater.

Fresh-machine install — the real story today

What the docs say

  1. Clone ~/projects/claude-sessions, bun install, bun run setup.
  2. Clone ~/.ccs-config.
  3. Run ccs sync-roles to materialize into ~/.claude/.

What's actually missing from that plan

  • Prereqs split by layer. Tool needs: bun, claude, cmux, git; optional codex. Cluster (pr-watch) needs: python3, gh, sf, jq. No unified install script + no per-layer check.
  • Config path. Docs assume ~/.ccs-config but there's no $CCS_CONFIG_ROOT env var enforcement.
  • LaunchAgent plists. Copy from cluster engine/deploy/ → ~/Library/LaunchAgents/ → sed /Users/mimen → $HOME → launchctl bootstrap. Per plist. No script.
  • Cluster engine dependency check. setup.ts checks tool-level deps (claude/codex/cmux) but nothing verifies the cluster's own deps (gh/sf/jq/python3 for pr-watch). Should live under ccs cluster doctor <name>.
  • Durable-state seed. No path to bring an existing PR queue to the new machine (or start fresh).
  • Version coherence. If machine 2 clones tool 5 commits ahead of machine 1, the cluster gate might refuse. No ccs doctor to check.

Realistic time estimate

  • 20 min for you, knowing the whole shape.
  • 2 hours for someone following the current README breadcrumb.
  • 1 command post-fix (once we have ccs install + ccs cluster install <name>).

P0 findings — block shareability

P0-1: 7 pr-watch Python scripts still open catalogue.db directly (ADR-0078 half-done)

approve.py, board.py, catalogue_sync.py, completion.py, mark_completed.py, pill_sweeper.py, retire_sweep.py all import sqlite3. Every catalogue schema change requires coordinating changes across two repos. Blocks ADR-0078's promise.

Fix: migrate each to ccs_client.catalogue_export(). ~2–3 hours.

P0-2: No versioned distribution artifact

ccs runs from source via a Bun symlink. Version is 0.1.0 forever. No git tags, no bun build --compile, no npm publish, no Homebrew. Sharing to another machine means "clone this git branch" — the branch, not a release.

Fix: add bun build --compile to release process. Bump pkg.version per release. Tag every release. Publish binaries to GitHub Releases. Optional: brew formula. ~1 day for the infrastructure, ~30 min per subsequent release.

P0-3: 11+ hardcoded /Users/mimen paths in LaunchAgent plists

Every plist under clusters/pr-watch/engine/deploy/ hardcodes the absolute path. On a fresh machine with $HOME=/Users/alice, launchd silently no-ops.

Fix: ship as .plist.template + install-launch-agents.sh that sed substitutes $HOME at install time. ~30 lines of shell + docs.

P0-4: No ccs cluster install for LaunchAgents

Installing the cluster's plists is a 3-step manual process (cp → sed → launchctl bootstrap) described only as XML comments inside each plist. No script does this.

Fix: ccs cluster install <name> that discovers plists in the cluster's engine/deploy/, templates $HOME, copies to ~/Library/LaunchAgents/, and bootstraps. Same command uninstalls on --remove.

Decisions to make

Six calls that shape the next 2–4 days of work.

D1 — Distribution strategy

Options:

  1. Bun compile → single binary → GitHub Releases. Preferred. Users don't need Bun; one file.
  2. npm publish + bunx. Simpler infra, but consumers still need Bun.
  3. Stay run-from-source. Acceptable only if the tool never leaves your machines.

My recommendation: option 1. It's a single bun build --compile line, and it produces a real versioned artifact you can point people at.

D2 — CLI reorganization scope

Options:

  1. Full noun-grouping now. ccs cluster/role/api/session/…. Break existing muscle memory in one shot; add aliases for a release.
  2. Additive nouns, keep flat. Add the nouns as new entry points; both work indefinitely. Slower discovery, but zero disruption.

My recommendation: option 1. Muscle memory heals; the flat namespace is a real usability tax on every new user.

D3 — TUI mutation strategy

Options:

  1. Shared mutations.ts. Both CLI's commands.ts and TUI call it. Setters in db.ts become module-private.
  2. TUI shells out to CLI. More disruptive; introduces subprocess overhead on every click.
  3. Status quo. Rely on convention that mutations get invariants added to setters.

My recommendation: option 1. Small refactor, huge invariant guarantee.

D4 — LaunchAgent installer

Options:

  1. Template + install script. Ship .plist.template, ccs cluster install does the sed + launchctl.
  2. Runtime $HOME resolution. Use launchctl setenv HOME "$HOME" before bootstrap; plists reference $HOME. More clever, less portable across users.

My recommendation: option 1. Standard practice, dead simple.

D5 — New-cluster wizard depth

Options:

  1. Minimal. cluster.toml + CHANGELOG.md only. User writes roles by hand.
  2. Standard. Above + one starter control role.
  3. Opinionated. Above + engine skeleton + LaunchAgent template + docs stubs.

My recommendation: option 2 initially, option 3 once ops-watch has clarified what a "real" cluster needs. Guard against baking pr-watch assumptions into the wizard prematurely.

D6 — Config repo separation

Question: should ~/.ccs-config stay a private per-user repo, or become a shared org-level repo with per-user overrides?

My recommendation: stay per-user for now. Sharing config is a separate problem from sharing the tool. Address after we have 2+ users.
