What this is

Markdown is hard to review, so this is the single navigable surface. Two things sit at the center:

  • The burn-down matrix — every concept (session resume, liveness, each pr-watch role, the phase machine, hooks, etc.) as a unit scored on six production axes: Live · Unit · Integration · Determinism · Hardened · Doc. Filter it, sort it, burn it down.
  • The glossary + system units — the shared vocabulary, then every unit of the system defined in dependency order (smallest primitives first, big flows last as compositions), each with its inputs and outputs. This is the mental model the rest of the page speaks in.
  • The P0 correctness cluster — the seven things that would actually bite in production, each verified against source with file:line.

Below those, each audit is a collapsible section preserving every finding, table, file:line, and failure scenario. Nothing was summarized away; the source is in docs/FINDINGS-*.md and docs/PRODUCTION-READINESS-CHECKLIST.md.

The short version (read this first)

Where it stands: ccs is solid at its core and works day-to-day — the liveness bridge, resume, and hook resolution are well-built and tested, and the whole pr-watch fleet runs on it. It is not yet production-hardened. The gap isn't "it's broken"; it's "a handful of edge cases could bite under load or after a dependency upgrade, and several things only work because we run them by hand rather than because a test proves they work."

The four themes that matter:

  1. A few correctness edges could double-spawn or corrupt state. The same PR can be claimed by two workers if two spawns race (no lock between "is it running?" and "start it"). The work-unit key that ties a session to its PR is copy-pasted six times and two copies drifted, so inbox routing can silently miss. These are rare-but-real, and they're the kind of bug that only shows up when the fleet is busy.
  2. The cmux dependency is unpinned. Everything hangs on the terminal (cmux) behaving exactly like version 0.64.17. Nothing checks the version, so a cmux upgrade could silently make ccs think every session is closed again — the exact bug we just fixed. There's also one old code path left that "fails open" (guesses nothing is running when it can't tell), which is the unsafe direction.
  3. The riskiest code has the least test coverage. The database migrations (which can corrupt the catalogue if wrong, and were hand-patched on the live DB), the spawn primitive, and the fail-closed safety check are all essentially untested — they work today because we watched them work, not because a test guards them.
  4. The whole pr-watch flow has never been proven end-to-end on a live fleet. The phase state machine, the tab pills, the review gate — all built, none confirmed working against a running worker.

What "done" looks like: fix the ~7 correctness edges (most are small — two are near one-liners), pin + version-check cmux, write tests for the 4 catastrophic-if-wrong paths, and drive one real PR through the fleet start-to-finish. Roughly a focused week of work. The burn-down order at the bottom sequences it.

The good news: zero dead files, near-zero commented-out code, strict typing, the cmux bridge is a model boundary, and the hooks pass their own linter. This is a maturity/hardening pass, not a rescue.

Glossary — the shared vocabulary

One canonical definition per concept, grounded in the actual code. The system-unit definitions below use only these terms, so the whole model reads in one language. Where a concept has two names in the code, the canonical term is defined and the alias is flagged as naming debt — using shared vocabulary everywhere is exactly what makes those inconsistencies visible. Full source: docs/GLOSSARY.md.

6 naming inconsistencies were surfaced by writing this glossary — all now decided (same concept, two names in the code). Resolved via ADR-0059 (naming reconciliation) + ADR-0057 (work-unit):
  • cluster — canonical everywhere; the DB column system is being renamed to cluster. (the big one, 44 files)
  • grouping — the platform concept; a grouping has a type, and pr-watch uses an epic-type grouping (not a rename — grouping is general, epic is a variant).
  • key — canonical; the deprecated event column/command is being removed.
  • role — canonical (ADR-0015); the deprecated skill column is being removed.
  • phase — removed entirely; superseded by stage × activity (renderers move fully off it).
  • work-unit — the 6-copy derived-string drift is retired by making the work-unit a first-class entity (ADR-0057), not by centralizing the string.
And one refinement: identity and responsibility are not synonyms (an earlier audit guessed they were) — a responsibility is the durable key; an identity is the runtime context it indexes.

Core identity

session — one Claude Code conversation (a transcript + its ccs metadata), named by a sessionId. A vessel: it can be closed and resumed; the identity it carries outlives it.

sessionId — the UUID that names a session (transcript filename, primary key).   resumeId — the UUID passed to claude --resume; equals sessionId for fresh sessions, can differ for old ones.

responsibility — the durable agent-identity KEY: {cluster?, role, epic?, workUnit?}. What a session is responsible for; survives resume.   identity — the runtime CONTEXT that key indexes (its inbox + state under ~/.ccs).

embodiment — a session actively running in a surface right now; liveness is the presence of an embodiment.   lineage — earlier embodiments of the same identity (matched by identity-key), so a fresh session can review prior attempts.

Work grouping

work-unit — the deliverable a session owns: pr:repo#number | gus:W-id | sid:sessionId. The unit of the one-embodiment rule.

cluster — a named set of sessions running one operation (e.g. pr-watch): the fleet + the core roles. (schema alias: system)   grouping — a mid-level work grouping bigger than a work-unit (a GUS epic); display metadata lives in cluster state. (schema/CLI alias: epic)

project — a loose user label.   key — an opaque identity-grouping slug (alias: event).   tag — an ad-hoc entity label.   parent / constellation — the optional session-to-session edge tree.

Role & kind

role — a session's first-class identity axis (control/concierge/slack-scout/eval/pr-agent/designer); drives which skills/commands/hooks materialize. (deprecated alias: skill)

kind — a role's class: loop (re-arms via resume_command, comes back running) or session (bare, rehydrates from inbox/state).   core vs fleet — singleton infra roles vs per-work-unit workers.

role-def — a role's definition read from files (role.toml + directory), never a DB cache: kind, homeDir, resumeCommand, skills[], commands[], hooks[].

Stores

catalogue — the durable metadata store (SQLite), one CatalogueRow per session; survives index rebuilds.   index — the ephemeral parsed-transcript cache (SQLite), one SessionRow per transcript; rebuildable.

state doc — a durable JSON file in an envelope {schemaVersion, updatedAt, source, data}, written atomically.   mergeFields — its single-writer-per-field update.   cluster state — state docs scoped to a cluster (board/gate/pending, grouping metadata).

inbox — a durable per-identity mailbox; delivery is independent of liveness.   message — one inbox item {path, sender, body}; drain is move-on-drain into processed/.

cmux embodiment

cmux — the terminal multiplexer sessions run inside (external binary, v0.64.17).   surface — one cmux pane where an agent runs (stable surfaceId).   workspace — a cmux tab holding surfaces; its tab is owned by the primary surface.

tree — the cmux JSON hierarchy (ground truth for what surface exists now).   hook store — cmux 0.64's file mapping surfaceId → sessionId (the authoritative binding).

bridge — the surface-keyed link between sessions and live cmux bodies, built by intersecting the hook store with the tree.   liveness — whether a session has a live surface (via the bridge, exact).   readable — whether the bridge's inputs were actually read; if not, spawning operations fail closed.

Spawn / resume

spawn — birth of a NEW session (mint id, write catalogue, launch).   spawnCmux — the ONE primitive that launches into a fresh detached workspace.   resume — revival of a CLOSED session (check liveness, else claude --resume).

spawn contract — the born-correct gate: refuse a worker spawn unless one-embodiment rule (≤1 live session per work-unit) and correct-worktree hold.   supersede-dedup — in a cluster resume, older dead siblings of a live work-unit are superseded, not resumed.   retired — completed/archived; never revived.   selector — a token resolving to sessionIds (id / #pr / W-id / grouping / role / cluster).

Hooks

hook — a layered config point firing at a Claude moment; each hook type (claude-md, start, stop, meta-update, cmux-paint, statusline, spawn-location, guard) declares a merge strategy.

level — one identity layer, ordered broad→specific: user → cluster → role → epic → work-unit → identity.   resolution — the pure map from a CatalogueRow to the ordered levels + config dirs.   merge — folding the per-level layers by the type's strategy.

start action — an action run at SessionStart: arm (re-fire a loop's resume_command) or drain-inbox.   meta-update — declares which CatalogueRow fields get refreshed at Stop (freshness contract, not an auto-writer).

Rendering & pr-watch phase

tab — a workspace's rendered cmux tab {title, description, color, statusPill}, synced from the row each turn by sync-tabs.   pill — the tab/statusline status indicator.   statusline — the single rendered status line.

stage — a PR's monotonic pipeline position (building→milad-review→in-review→approved→merged).   activity — the transient state within a stage (working/needs-you/fixing). (these replaced the single phase column)   buildComplete — the latch sealing the build stage once milad-review is hit.

gate — the invariant that a PR clears internal + Milad review before public review.   miladReview — Milad's +1 verdict the gate reads.   sense — the engine step writing git/GitHub/Slack facts to cluster state.   mark — the control-owned completed/archived lifecycle transition.

Board publish surface (ADR-0077)

board.json — the cluster-provided publish surface: identity-keyed rows with render primitives (pills, alerts, description, awaitingFrom) plus opaque data. Tool defines schema, cluster owns vocabulary.

BoardRow — one identity's composed state: {identity, workUnit, sessions[], pills[], description, alerts[], awaitingFrom[], lastComposed, data}. Keyed on identity, not sessionId, so twins share one row.

Pill — a render primitive: {key, label, icon?, color?, priority?}. The tool paints as-is; cluster owns the label/color vocabulary (e.g. "in review", "#bf5af2").

Alert — a named sensor-backed signal: {name, severity, reason, owner, sinceTick?}. Cluster emits; tool renders (as badge/pill); neither interprets the other's vocabulary.

awaitingFrom — multiset of actor labels (cluster vocabulary, e.g. ["reviewers", "worker"]) — who must act next. Tool renders as text; doesn't parse.

identity — the durable axis: cluster:role:work-unit. Board rows key on this; sessions resolve to it; survives respawn/twin churn.

freshness contract — writes that change composed state trigger synchronous recompose before returning, so consumers see new state immediately. Enforced by recomposeForSession calls after meta-set / stage-set / sync-tabs.

pidAlive — liveness predicate: process.kill(pid, 0) honest check. Filters hook-store bindings so dead-pid phantoms don't count as live.

reap-duplicates — ccs reap-duplicates: find sessions with >1 live claude proc (via ps), rank by CPU time, close duplicates. Dry-run default.
