# cmux 0.64 moved session tracking to a hook store — the bridge repoints, and liveness fails closed

cmux updated to **0.64.17** and changed how it tracks Claude sessions. That silently broke ccs
liveness: every session read as closed, and `ccs resume` would have re-spawned the entire running
fleet (fail-open duplication). Verified live 2026-07-11 by a designer session spawned through the
fixed path, which is itself the proof the fix works. Supersedes the persisted-state join of
ADR-0040/0042 and hardens the resume contract of ADR-0015.

## What broke (observed)

- `cmux tree` and `ccs whoami` still worked, but `ccs resume-cluster pr-watch --dry-run` reported
  `0 already open` for a fleet that was demonstrably running. Two symptoms, one cause:
  1. ccs showed EVERYTHING closed.
  2. A real `ccs resume` would treat "closed" as "spawn it" → a duplicate pane per live PR worker
     → the duplicate-fleet runaway we already feared (ADR-0015 supersede-dedup assumes readable
     liveness).

## Why — the verified mechanism (cmux 0.64)

- **OLD (≤0.64.10, ADR-0040):** cmux persisted `agent.sessionId` per panel in the app-state file
  `~/Library/Application Support/cmux/session-com.cmuxterm.app.json`. ccs joined that file's panel
  UUID against `cmux tree`'s surface UUID (25/25 overlap) to resolve liveness.
- **NEW (0.64.17):** the app-state file no longer records `agent.sessionId` (a live read found only
  2 stale leftover panels). Session identity now lives in a hook-populated store,
  **`~/.cmuxterm/claude-hook-sessions.json`**:
  - `sessions[sessionId] = {surfaceId, workspaceId, pid, cwd, transcriptPath, agentLifecycle,
    isRestorable, startedAt, updatedAt, launchCommand}` — per-session detail. **Accretes history:**
    dead and replaced sessions stay in the map (a live read showed 12 entries for ~5 live surfaces).
  - `activeSessionsBySurface[surfaceUUID] = {sessionId, …}` — the CURRENT surface→session binding.
    This is the authoritative join key. Also accretes: it held 9 bindings when only 2 of those
    surfaces still existed in `cmux tree`.
- The store is populated **only by cmux's `claude` shim hooks**, which fire **only when `claude` is
  launched as a plain command in a cmux integrated shell**. A bare `exec claude` or an
  env-scrubbed spawn bypasses the shim and never registers (this is why the running pre-fix fleet
  is invisible — see Consequences).
- The cmux socket requires AUTH (`CMUX_SOCKET_PASSWORD` / integration-provided). An unauthed shell
  gets "broken pipe, errno 32" on every `cmux` call — indistinguishable at the data layer from
  "nothing open" unless we track readability explicitly.

## Decision

1. **Repoint the bridge to the hook store.** `src/cmux/bridge.ts` parses
   `activeSessionsBySurface` (authoritative current binding) enriched from `sessions` (detail),
   replacing the dead `agent.sessionId` parse. `src/cmux/live.ts` reads
   `~/.cmuxterm/claude-hook-sessions.json` instead of the app-state file.

2. **Liveness = store ∩ tree.** Because the store accretes stale bindings, a binding counts as
   live ONLY if its surface still exists in `cmux tree` (the ground truth for "what exists now").
   `buildBridge` already intersected on the tree; keeping that is what makes the accretion
   harmless. `cmux tree` remains the ordering source for primary-surface / tab ownership
   (ADR-0027/0040), unchanged.

3. **Readability is a first-class result, and resume FAILS CLOSED on it.** The bridge carries a
   `readable` flag: true only if BOTH `cmux tree` succeeded AND the hook store was
   present-and-parseable. A missing store file is readable-but-empty (cmux installed, nothing
   tracked yet); a failed `cmux tree` (binary absent, socket unauthed, non-zero exit) or an
   unparseable store is UNREADABLE. When liveness is unreadable, `ccs resume-session`,
   `ccs resume-cluster`, and `ccs resume <selector>` ABORT and spawn nothing — never treat
   unreadable as "closed". This is the real defect behind the duplicate-fleet runaway: it is
   always safer to skip a resume than to duplicate a running fleet.

4. **Spawn stays a plain command (do NOT scrub CMUX_SURFACE_ID) — reconciling ADR-0042.** ADR-0042
   scrubbed `CMUX_SURFACE_ID`/`CMUX_WORKSPACE_ID` to stop an INLINE spawn from inheriting the
   caller's surface and hijacking its binding. But the 0.64 shim NEEDS those vars to register the
   session, and we already spawn into a FRESH `cmux new-workspace` (own surface id, nothing to
   hijack, one-session-per-workspace so no sibling to clobber). So the scrub was both unnecessary
   here and actively harmful: it produced untracked sessions. `spawn-cmux.ts` runs a plain command,
   no `exec`, no scrub. ADR-0042's hijack guard still holds for its actual case (never inline-launch
   into the caller's terminal); the scrub specifically is retired for the detached path.

## Consequences

- The already-running pr-watch fleet (launched the OLD bare way, pre-fix) is NOT in the hook store,
  so ccs sees it as closed. It is NOT unreadable (the store reads fine), so the fail-closed guard
  does NOT protect against re-spawning it — a blind `ccs resume-cluster pr-watch` right now would
  duplicate ~18 live workers. **Mitigation until every worker restarts naturally through the fixed
  spawn path: do not run a broad cluster resume against the pre-fix fleet.** New spawns register
  correctly, so this is self-healing as workers turn over. (Constitution already bars agents from
  resuming the cluster; resume is Milad's explicit action or the scheduler.)
- Consumers are untouched: they go through the stable `Bridge` interface. The persisted-file
  parser/type (`parsePersisted`/`CmuxPersisted`/`PersistedAgent`) and the `persisted.json` fixture
  are deleted; a `hook-store.json` fixture (25 live bindings + 2 deliberately-stale) replaces them.
- `surfaceInfo` now returns a `SurfaceSession` (sessionId + workspace/cwd/lifecycle/restorable).
  Production only reads `.sessionId`; the resume-binding replay is gone (the store doesn't carry a
  resume command, and ccs derives its own launch dir + resume command from the catalogue anyway —
  ADR-0021/0042).
